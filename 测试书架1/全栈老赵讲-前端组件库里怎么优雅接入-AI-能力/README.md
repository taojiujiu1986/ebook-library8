# 为什么前端组件库要组件化 AI 能力

## 开篇引入：AI 能力为什么不能只写在页面里

一句话回答：**因为 AI 功能一旦只写在业务页面里，就会把交互、状态、错误处理和样式规范都散落到各处，最终变成“能用但难复用、难维护、难测试”的一次性代码。**

很多团队第一次接入 AI，通常是“哪里要用，哪里写一段”：表单页加一个“智能润色”按钮，详情页放一个“AI 建议”面板，弹窗里再临时拼一个“生成文案”逻辑。看起来快，实际上只是把复杂度从后端接口转移到了前端页面里：交互不统一、loading 和 disabled 各写各的、错误提示不一致、重试逻辑分散、埋点遗漏，最后测试也只能靠人工反复点。

本书的出发点很明确：**把 AI 能力当成前端可复用资产，而不是一次性页面代码。**也就是说，要站在组件库和工程化角度，把 AI 功能抽象成组件或 Hook，让它能在不同业务页面中稳定复用。这样做不是为了少写几行代码，而是让 AI 功能具备可维护、可替换、可测试、可演进的工程属性。

> 边界先说清：本书只讲前端工程落地，不展开大模型原理，也不讨论模型底层机制。我们关注的是如何把 AI 能力做成组件、Hook 和规范，而不是解释“模型为什么会生成”。

## 业务页面中 AI 能力重复开发的典型问题

### 为什么“临时接入”很快就会失控？

如果每个页面都自己写 AI 交互，通常会很快暴露几类问题。

第一是**交互重复**。同样是“点击生成、显示 loading、返回结果、允许重试”的流程，在表单页、详情页、弹窗页可能写三份。表面上是 UI 不同，本质却是同一套状态机，没必要重复造轮子。

第二是**状态分散**。一个页面里往往同时存在输入状态、请求状态、结果状态、禁用态、错误态、空态。若这些状态散落在页面组件里，代码会很快变成一团：按钮能不能点、结果要不要覆盖原文、失败后是否允许继续提交，这些规则很难一眼看清。

第三是**样式不一致**。同样是 AI 文案生成，有的页面用骨架屏，有的页面用转圈 loading，有的页面用“生成中，请稍等”，用户会觉得这是三个不同能力，而不是同一产品体系里的功能。

第四是**可测试性差**。请求逻辑直接写在页面中，单元测试要同时模拟输入、点击、异步返回、失败重试，页面一改，测试就跟着大面积失效。

根源不在于 AI 功能复杂，而在于它没有被抽象成标准化交互能力。

## 把 AI 能力做成组件/Hook 的价值是什么？

### 统一沉淀，分别解决什么问题？

把 AI 功能抽成组件或 Hook，不只是为了少写代码，而是为了把能力沉淀成工程资产。

- **复用**：`AI输入框`、`AI建议面板`、`AI文本润色按钮`、`AI生成弹窗`、`AI结果卡片` 可以在多个业务页面直接复用。
- **统一**：统一 `loading`、`disabled`、`error`、`empty`、`success` 等状态表现，避免体验割裂。
- **可维护**：请求逻辑、错误兜底、重试机制集中封装，后续改接口只需要改一处。
- **可测试**：组件负责展示，Hook 负责状态机和异步流程，测试边界更清晰。
- **可演进**：后续要加埋点、权限控制、缓存、节流、二次确认，都能在统一抽象上扩展。

### 组件还是 Hook？按什么标准选？

简单判断标准是：**视觉强相关的做组件，逻辑可复用的做 Hook。**

比如：

- `AI输入框`：适合做组件，负责输入区布局、触发按钮、提示文案和交互反馈。
- `AI建议面板`：适合做组件，负责建议项展示、选择、复制、再次生成。
- `AI文本润色按钮`：适合做轻量组件，直接挂在文本编辑器旁边。
- `AI生成弹窗`：适合做组件，承载参数配置、生成预览、确认提交。
- `AI结果卡片`：适合做组件，统一展示结果内容、操作按钮、反馈入口。

更适合放进 Hook 的能力包括：

- 请求发起与取消
- 状态机管理
- 重试与节流
- 错误归一化
- 结果缓存
- loading 和 disabled 的切换

常见原则是：**把变化大、组合强的逻辑放到 Hook，把视觉和交互入口做成组件。**这样既保留灵活性，又不让页面承担过多业务细节。

## 组件/Hook 设计时，先统一什么？

### 先统一状态拆分和状态机

AI 组件最容易失控的地方不是按钮或弹窗本身，而是状态。建议先把状态拆成三层：

- **输入态**：`value`、`defaultValue`、`placeholder`
- **请求态**：`idle`、`loading`、`success`、`error`
- **展示态**：`result`、`empty`、`disabled`

例如 `AI生成弹窗` 可以定义清晰状态机：

| 状态 | 说明 | UI 表现 |
| --- | --- | --- |
| idle | 未触发生成 | 展示配置区和确认按钮 |
| loading | 请求中 | 按钮禁用，显示 loading |
| success | 生成成功 | 展示结果预览和操作区 |
| error | 生成失败 | 展示错误提示和重试按钮 |

这样页面接入时，不需要猜“这个按钮现在能不能点”，只要看状态机就能判断。

### props、事件、children 怎么设计

一个可复用组件，最怕 props 零散、事件随意、插槽没边界。建议保持统一命名。

以 `AI建议面板` 为例，可以这样设计：

- `props`
  - `items`: 建议列表
  - `loading`: 是否请求中
  - `disabled`: 是否禁用
  - `emptyText`: 空态文案
  - `errorText`: 错误文案
- `事件`
  - `onSelect(item)`: 选中建议
  - `onRetry()`: 重新生成
  - `onCopy(item)`: 复制建议
- `children / 插槽`
  - `header`: 自定义标题区
  - `footer`: 自定义底部操作区
  - `empty`: 自定义空态
  - `actions`: 自定义按钮组

同样，`AI结果卡片` 也可以通过 `actions` children 注入“替换原文”“复制结果”“再次生成”等操作，而不是把所有按钮都写死在组件内部。

### Hook 如何配合组件一起用

`useAIAction` 这类 Hook 可以负责：

- 组织请求参数
- 控制 `loading` 和 `disabled`
- 管理 `idle -> loading -> success/error` 状态机
- 统一错误归一化
- 暴露 `run / retry / cancel`

组件只关心怎么展示，Hook 只关心怎么跑流程。这样页面接入时，通常只需要传入 `prompt`、`params`、`onSuccess`、`onError` 等配置项即可。

## 从一次性功能到平台化能力，怎么演进？

### 先页面内抽离，再组件化沉淀

不要一上来就追求“万能 AI 组件库”。更稳妥的路径是三步走。

1. **页面内抽离**：先把重复 AI 请求逻辑提成 Hook。
2. **组件化入口**：再把高频交互封装成 `AI输入框`、`AI文本润色按钮`、`AI生成弹窗`、`AI建议面板`、`AI结果卡片`。
3. **沉淀规范**：最后统一 props、事件、children、loading、disabled、错误兜底和视觉一致性。

例如，`AI文本润色按钮` 可以暴露 `text`、`loading`、`disabled`、`onGenerate`、`onApply`、`onRetry` 等 props；`AI生成弹窗` 可以额外接收 `title`、`mode`、`defaultParams`、`confirmText`、`cancelText`，让不同业务页只改配置，不改核心逻辑。

## 本书的范围边界：只讲前端工程落地

### 重点关注什么

本书不会展开大模型原理。我们关注的是前端开发者真正要解决的问题：

- 如何设计可复用的 AI 组件 API
- 如何拆分状态和异步交互
- 如何处理加载、禁用、错误兜底
- 如何保持视觉一致性
- 如何提升可测试性与文档化质量
- 如何把组件发布、维护并在多个业务页面复用

换句话说，这本书的目标不是教你“模型为什么会生成”，而是教你“前端如何把 AI 能力做成稳定、可维护、可复用的工程资产”。

## 结尾小结：先统一抽象，再谈快速落地

如果你现在正在把 AI 接入组件库，最重要的不是先堆功能，而是先回答三个问题：

1. 这个能力是页面专属，还是可以复用？
2. 它的状态机是否清晰，是否能统一处理 `loading`、`disabled`、`error`？
3. 它能否通过组件或 Hook 被其他页面直接使用？

当你开始用这三个问题审视 AI 功能时，组件化的方向就已经明确了。后续我们会继续拆解：如何设计 API、如何封装 Hook、如何处理状态、错误和文档，让 AI 能力真正沉淀为前端可复用资产。

# 先抽象再实现：AI 交互的通用模式拆解

## 开篇引入：先把“AI 功能”当成交互，而不是特例

前端一接入 AI，最常见的写法就是“哪里需要就写哪里”：页面里塞一个输入框、一个请求函数、一个加载态，再补几句错误提示。短期能跑，长期一扩展就失控：同样的润色按钮在编辑页、详情页、弹窗里各写一遍；同样的生成流程有三套状态；同样的失败重试逻辑散落在多个文件里。

问题不在于 AI 功能复杂，而在于我们把它当成了“特殊能力”，没有把它看成一种可标准化的前端交互。只要思路一变，事情就清楚了：AI 不是某个页面的临时逻辑，而是可以抽成组件、Hook 和状态机的一类通用能力。

本章不讲大模型原理，只讲如何把 AI 功能抽象成**可复用的前端交互模式**。一旦抽象正确，后面无论是做 **AI 输入框**、**AI 建议面板**、**AI 文本润色按钮**、**AI 生成弹窗**，还是 **AI 结果卡片**，都可以复用同一套状态、事件和职责划分。

核心原则很简单：**先抽象交互，再实现组件；先定义边界，再写 UI。**

## 一、从一个统一案例看五类 AI 交互怎么串起来

为了避免每个组件各讲一套，先统一一个案例：**商品标题优化**。

这个案例里，前端会经历完整链路：

1. 用户在 **AI 输入框** 里输入商品卖点；
2. 系统给出若干 **AI 建议面板** 候选标题；
3. 用户点击 **AI 文本润色按钮** 对当前标题做精修；
4. 通过 **AI 生成弹窗** 选择风格、字数、关键词后生成新标题；
5. 最终把结果放进 **AI 结果卡片**，供用户复制、对比、应用。

### 1. 五类交互的职责边界

- **AI 输入框**：收集上下文，发起交互，不负责结果渲染
- **AI 建议面板**：展示候选项，支持选择、刷新、忽略
- **AI 文本润色按钮**：围绕已有文本触发优化，关注触发和应用
- **AI 生成弹窗**：承载参数输入和长流程确认，关注流程控制
- **AI 结果卡片**：承载结果展示、复制、重试、反馈等动作

这五类能力本质都在处理同一个问题：**如何把一次 AI 交互做成可配置、可复用、可测试的前端资产。**

### 2. 五类场景的共同链路

它们逃不开同一条链路：

- **触发**：点击按钮、输入文本、打开弹窗、切换选项
- **请求**：整理参数并发起异步调用
- **加载**：展示 loading，避免重复触发
- **成功**：渲染结果，提供应用或复制动作
- **失败**：保留上下文，给出错误信息
- **重试**：沿用原参数重新请求
- **结束**：应用结果或关闭交互，进入下一步

也就是说，五类组件只是“外壳不同”，底层流程是同一套。抽象的目标不是抹平差异，而是把共性收拢到可复用模板里。

## 二、统一状态机：所有 AI 交互都先过这一张表

如果后续要写组件 API，先别急着定 props，先把状态机定下来。状态不统一，组件交互就会乱。

| 状态 | 含义 | 典型 UI | 可执行动作 |
| --- | --- | --- | --- |
| `idle` | 初始空闲 | 默认展示、等待输入 | 输入、点击、打开 |
| `loading` | 请求中 | 加载提示、禁用重复触发 | 取消或等待 |
| `success` | 请求成功 | 展示结果、可应用 | 复制、确认、继续编辑 |
| `error` | 请求失败 | 错误提示、保留上下文 | 重试、关闭 |
| `retrying` | 正在重试 | 显示重试中 | 等待结果 |
| `done` | 已完成应用 | 标记已应用、收起结果 | 开启下一轮 |

这张表的价值有三个：

1. **统一 props 命名**：`loading`、`error`、`value`、`result` 一看就知道含义；
2. **统一事件命名**：`onSubmit`、`onRetry`、`onConfirm`、`onCancel` 可以在各组件复用；
3. **统一样式规范**：加载态、失败态、完成态都能按同一套视觉规则实现。

### 状态拆分要避免“一把梭”

不要把所有情况都塞成一个 `status` 字符串，也不要只靠一个 `isLoading` 管全流程。更合理的拆法是：

- **业务数据状态**：`prompt`、`result`、`errorMessage`
- **UI 状态**：`open`、`loading`、`disabled`、`expanded`
- **交互状态**：`selectedId`、`hasApplied`、`retryCount`

比如 **AI 建议面板**，可能同时存在“有建议但未选中”“已选中但未应用”“上次失败但保留旧列表”这些中间态。状态拆得清楚，组件就不会在复杂场景下失真。

## 三、组件职责边界：组件管表现，Hook 管流程，页面管业务

### 1. 哪些逻辑该放组件里，哪些该下沉到 Hook

抽象时最怕职责混乱。建议直接按三层切：

#### 组件负责
- 展示 UI
- 接收 props
- 触发事件
- 呈现状态
- 提供 `children` / 插槽扩展位

#### Hook 负责
- 管理状态机
- 处理异步请求
- 统一错误处理
- 组织重试逻辑
- 输出可供 UI 消费的状态与动作

#### 页面层负责
- 业务上下文拼装
- 接口参数映射
- 埋点、权限、路由联动
- 最终应用结果后的刷新或保存

最稳的边界就是：**组件管表现，Hook 管流程，页面管业务。**

### 2. 一个可复用的 Hook 封装模板

可以把通用链路收敛成一个 Hook，例如 `useAIFlow`：

```ts
type AIStatus = 'idle' | 'loading' | 'success' | 'error' | 'retrying' | 'done'

interface UseAIFlowOptions<TParams, TResult> {
  requester: (params: TParams) => Promise<TResult>
  defaultParams?: Partial<TParams>
  autoRun?: boolean
}

interface UseAIFlowReturn<TParams, TResult> {
  status: AIStatus
  loading: boolean
  data: TResult | null
  error: string | null
  run: (params?: Partial<TParams>) => Promise<void>
  retry: () => Promise<void>
  reset: () => void
  cancel: () => void
}
```

它的意义在于：**AI 输入框**、**AI 建议面板**、**AI 文本润色按钮**、**AI 生成弹窗**、**AI 结果卡片**都能共用同一个流程骨架，只替换参数和渲染层。

### 3. props、事件、children 的统一写法

以“AI 生成弹窗”为例，推荐的 API 边界可以这样定义：

- `open`：是否打开
- `title`：弹窗标题
- `prompt`：输入内容
- `loading`：是否请求中
- `result`：生成结果
- `error`：错误信息
- `mode`：`'replace' | 'append' | 'insert'`
- `onSubmit(prompt)`：提交生成
- `onRetry()`：重试请求
- `onConfirm(result)`：确认应用
- `onCancel()`：关闭弹窗
- `children`：自定义底部操作区或结果区

这里要特别强调：**props 描述状态，events 描述动作，children 描述扩展**。把这三类边界定清楚，后面换场景时只换壳，不换心。

## 四、样式、加载与禁用态：让组件像一套系统

### 1. 视觉一致性不能靠各页面自己调

AI 组件进入多个页面后，最容易发散的是视觉：按钮大小不一、错误颜色不一、加载动画不一、空态文案不一。解决方式不是“每页修一下”，而是定义统一规范：

- `loading`：统一使用骨架屏、转圈或局部遮罩
- `disabled`：统一降低对比度并禁止交互
- `error`：统一错误图标、错误文案和重试入口
- `success`：统一结果高亮方式
- `empty`：统一空态提示和引导动作

### 2. 加载与禁用态必须前置设计

很多 AI 交互的问题都出在这儿：用户连点两次、请求还没完就又发一次、弹窗关闭后还在回调、结果区被旧数据覆盖。正确做法是：

- 请求中禁用触发按钮
- 输入框在必要时只读
- 重试时保留原始参数
- 成功后再释放确认按钮
- 结束态明确标记已应用

所以 **加载与禁用态** 不是样式细节，而是状态机的一部分。

---

## 五、错误态和兜底策略：别让失败只剩一句“请求失败”

AI 交互一定会遇到失败，所以错误态不是附属功能，而是组件设计的一部分。

建议至少提供三层信息：

- `message`：用户可见的错误提示
- `details`：可选的技术原因或调试信息
- `retry`：重新发起请求的动作

在组件 API 上，可以这样设计：

- `error`：错误对象或错误文案
- `errorRender`：自定义错误展示
- `onRetry`：重试事件
- `fallback`：兜底内容

例如 **AI 结果卡片** 在失败时，不要直接清空原结果，而是保留上一次成功内容，显示一个“重新生成”按钮。这样用户既能理解失败，也不会丢掉上下文。对于 **AI 文本润色按钮**，失败时则更适合只回退按钮状态，不影响编辑器正文。

---

## 六、组件文档示例：把抽象讲清楚，才能真正复用

文档化不是附属工作，而是组件化资产的一部分。建议每个 AI 组件都固定输出一份 README 或文档页模板。

### 1. 推荐文档结构

### 2. 文档里要把示例写全

至少要覆盖：

文档越接近真实用法，复用成本越低。否则组件虽然做出来了，但团队还是会重复抄代码。

## 七、五类 AI 场景如何复用同一套模板

把前面的抽象落到具体场景里，会更直观。

### 1. AI 输入框

适合收集上下文并发起请求，比如用户输入商品卖点、目标人群、语气风格。它通常暴露：

- `value`
- `placeholder`
- `loading`
- `disabled`
- `onChange`
- `onSubmit`
- `children`

输入框本身不决定结果怎么展示，只负责触发和承接上下文。

### 2. AI 建议面板

适合展示多个候选项，例如标题建议、摘要建议、标签建议。它通常需要：

- `items`
- `selectedId`
- `loading`
- `error`
- `onSelect`
- `onRefresh`
- `onIgnore`
- `children`

建议面板最重要的是“可选、可刷、可忽略”，而不是把所有建议一次性塞满。

### 3. AI 文本润色按钮

适合在已有文本上做局部优化。它通常只关心：

- `text`
- `loading`
- `disabled`
- `mode`
- `onClick`
- `onRetry`
- `onApply`

这种组件非常适合用 Hook 封装请求流程，再把按钮作为轻量表现层。

### 4. AI 生成弹窗

适合长流程、带参数配置的交互。它通常包含：

- `open`
- `params`
- `loading`
- `result`
- `error`
- `onSubmit`
- `onConfirm`
- `onCancel`
- `children`

它的优势是把“生成前配置”和“生成后确认”收拢在一个界面里，减少页面跳转。

### 5. AI 结果卡片

适合承接最终结果，支持复制、对比、应用和重新生成。它通常暴露：

- `result`
- `loading`
- `error`
- `applied`
- `onCopy`
- `onApply`
- `onRetry`
- `children`

结果卡片的关键不是“展示一段文本”，而是把结果变成可操作的业务资产。

---

## 八、在多个业务页面复用：同一套能力如何换壳不换心

### 1. 统一案例的复用路径

继续用“商品标题优化”这个案例：

- 在商品编辑页，AI 输入框用于收集标题关键词；
- 在列表页，AI 建议面板用于批量给出优化建议；
- 在详情页，AI 文本润色按钮直接优化当前标题；
- 在配置弹窗里，AI 生成弹窗负责参数输入与确认；
- 在历史记录页，AI 结果卡片负责展示生成结果并支持复制。

页面看起来不同，但底层共用同一套状态机、同一套 Hook、同一套错误兜底和加载规范。

### 2. 复用时最容易变化的是什么

通常变化的是：

- 接口参数
- 结果展示方式
- 文案
- 是否允许外部控制
- 是否需要 `children` 扩展

通常不变的是：

- 请求状态流程
- 加载/禁用规则
- 错误处理
- 重试逻辑
- 事件命名规范

这就是为什么 AI 能力适合抽成组件和 Hook：**变化集中在配置，不变化沉淀为资产。**

---

## 九、可测试性、发布与维护：让 AI 组件能长期活下去

### 1. 可测试性怎么设计

AI 组件最好从一开始就支持测试，尤其是以下场景：

- `loading` 时按钮是否禁用
- `error` 时是否显示重试
- `success` 时是否正确渲染结果
- `onSubmit`、`onRetry`、`onConfirm` 是否触发正确
- `children` 是否能被正确替换

建议把测试拆成三类：

- **单元测试**：Hook 状态机转移
- **组件测试**：props、事件、插槽渲染
- **交互测试**：从触发到成功/失败/重试的完整链路

这样做的好处是，状态机一变，测试能第一时间告诉你哪些组件受影响。

### 2. 发布与维护要有版本策略

AI 组件一旦被多个页面依赖，就不能随意改接口。发布时建议遵守：

- props 改动尽量兼容旧版本
- 新增能力优先走可选项
- 删除字段前给迁移期
- 状态机变化要同步文档
- 失败态、重试态文案要统一审核

如果要升级 `AI 结果卡片` 的结构，最好同时给出：

- 新旧 props 对照
- 新旧事件对照
- `children` / 插槽迁移方式
- 测试用例更新清单

这样组件才能从“能用”变成“可持续维护”。

---

## 结尾小结：先统一模式，再写具体组件

本章的核心只有一句话：**AI 功能不是一个个孤立能力，而是一组可以标准化的前端交互模式。**

你只要先把“输入、建议、润色、生成、结果展示”这些形态拆清楚，再把“触发、请求、加载、成功、失败、重试、结束”这些链路统一起来，就能自然过渡到状态机和组件 API 设计。接下来无论是做 **AI 输入框**、**AI 建议面板**、**AI 文本润色按钮**、**AI 生成弹窗** 还是 **AI 结果卡片**，都可以复用同一套思路。

### 这一章最重要的三件事

1. 先按交互形态抽象，再决定组件边界  
2. 用状态机管理 `idle/loading/success/error/retrying/done`  
3. 把 props、事件、`children`/插槽、Hook、错误兜底做成统一规范  

### 直接可执行的下一步

- 挑一个现有 AI 场景，画出状态机
- 标出哪些逻辑放进 Hook，哪些留在组件层
- 给组件补齐 props 表、事件表、插槽说明和错误态说明
- 用同一套模板复用到第二个页面，验证是否真的“换壳不换心”

只有先抽象，后实现，AI 能力才会真正变成可复用的工程资产。

# 第3章 AI 输入框：最小可复用单元怎么设计

# AI 输入框：最小可复用单元怎么设计

## 开篇引入：为什么先从 AI 输入框开始

在前端组件库里接入 AI 能力，最容易做散的就是输入交互。很多页面都有相似流程：输入、选择模式、提交、等待、失败重试。  
如果每个业务页各写一套，会直接带来三类问题：

1. **交互不统一**：回车提交、按钮提交、取消、重试规则各不相同。
2. **状态不一致**：加载、禁用、错误表现割裂，用户感知混乱。
3. **能力不可复用**：AI 能力没有沉淀成组件或 Hook，后面做“建议面板”“文本润色”“生成弹窗”还要重写。

所以，AI 输入框不是普通表单控件，而是**最小可复用单元**：它承载输入、提交、等待、反馈、恢复的完整链路。先把它设计好，后面的 AI 建议面板、AI 文本润色按钮、AI 生成弹窗、AI 结果卡片，才能复用同一套逻辑。

从工程角度看，先做输入框还有一个好处：它能先抽出 AI 交互里最常见的骨架，再向上叠业务结果展示、模板选择、上下文注入，组件体系不容易一开始就发散。

---

## 一、AI 输入框的能力边界与典型使用场景

### 1. 它应该解决什么问题？

在本书语境里，AI 输入框负责的是“**收集用户意图并触发 AI 任务**”，而不是负责生成结果本身。边界要清楚：

- 负责输入文本、辅助提示、提交请求
- 负责加载、禁用、错误、重试、取消
- 不直接耦合具体业务结果展示
- 不把模型参数写死在组件内部

也就是说，组件负责交互，Hook 或业务层负责任务执行。请求体怎么拼、返回结果怎么归类、失败后是否走兜底文案，都更适合放在外层封装。

### 2. 典型场景

- **AI 建议面板**：输入问题，得到建议列表或推荐文案
- **AI 文本润色按钮**：在输入框旁提供“润色一下”
- **AI 生成弹窗**：在弹窗里输入关键词后生成内容
- **AI 结果卡片**：输入后生成结构化结果并展示

这些场景表面不同，本质都需要一套输入、触发、等待、恢复的标准流程。把 AI 输入框抽成公共能力后，就能在多个业务页直接拼装，而不是每个页面重新写状态判断。

### 3. 设计原则

- **单一职责**：只做输入与提交，不吞业务逻辑
- **可配置**：通过 props 配置行为，而不是复制组件
- **可组合**：通过 children/插槽扩展前缀、后缀、操作区
- **状态显式**：不要用一个 `loading` 包打天下，要拆清楚
- **可测试**：每个状态都能独立触发和断言

---

## 二、组件 API 设计：先把 props 和事件定稳

一个可复用的 AI 输入框，先别急着加花活，最重要的是把 `props`、`events` 和可插拔区域定成稳定接口。

### 1. props 设计表

| 字段 | 类型 | 默认值 | 是否必填 | 作用 | 使用场景 |
|---|---|---:|---|---|---|
| `value` | `string` | `''` | 否 | 受控输入值 | 需要外部统一管理输入内容 |
| `placeholder` | `string` | `请输入内容` | 否 | 输入引导文案 | 不同业务语义提示 |
| `loading` | `boolean` | `false` | 否 | 是否提交中 | 请求未结束时禁用部分操作 |
| `disabled` | `boolean` | `false` | 否 | 是否整体禁用 | 页面锁定、权限不足、全局停用 |
| `mode` | `'chat' \| 'compose' \| 'rewrite'` | `'chat'` | 否 | 决定交互模式 | 聊天、创作、润色等不同语义 |
| `config` | `object` | `{}` | 否 | 可配置项集合 | 控制字数、快捷键、重试等 |
| `autoFocus` | `boolean` | `false` | 否 | 是否自动聚焦 | 弹窗内输入、首屏快速开始 |
| `readonly` | `boolean` | `false` | 否 | 是否只读 | 查看历史输入、提交中锁定输入区 |

### 2. config 应该装什么

不要把所有开关都摊成顶层 props，建议收敛进 `config`：

```tsx
<AIInput
  config={{
    maxLength: 200,
    submitOnEnter: true,
    allowEmptySubmit: false,
    showCounter: true,
    retryable: true,
    cancelable: true,
    showShortcutHint: true
  }}
/>
```

常见可配置项包括：

- `maxLength`
- `submitOnEnter`
- `allowEmptySubmit`
- `showShortcutHint`
- `showCounter`
- `retryable`
- `cancelable`

这样做的好处是 API 更干净，扩展更平滑。后续要加“历史输入”“模板选择”“敏感词提示”，继续往 `config` 里加即可。

### 3. 事件设计表

```tsx
<AIInput
  onChange={(v) => setText(v)}
  onSubmit={(payload) => submitPrompt(payload)}
  onRetry={() => retryLastRequest()}
  onCancel={() => abortRequest()}
/>
```

| 事件 | 触发时机 | 参数签名 | 典型用途 |
|---|---|---|---|
| `onChange` | 输入内容变化 | `(value: string) => void` | 同步受控值、实时校验 |
| `onSubmit` | 点击提交或回车提交 | `(payload: { value: string; mode: string; config?: Record<string, unknown> }) => void` | 发起 AI 请求 |
| `onRetry` | 错误态点击重试 | `({ value, lastError, context }: { value: string; lastError?: Error; context?: unknown }) => void` | 复用上次参数重新请求 |
| `onCancel` | 提交中点击取消 | `({ requestId?, reason? }: { requestId?: string; reason?: string }) => void` | 中止当前任务、恢复编辑态 |

### 4. 事件设计要点

- `onSubmit` 要尽量携带完整上下文，但不要替业务层拼请求体
- `onRetry` 不是可有可无，它决定错误态能否真正闭环
- `onCancel` 要保留，尤其适合长耗时生成、弹窗关闭和中断请求
- `onChange` 负责状态同步，别把校验、请求、清空逻辑都塞进去

一个常见误区是把 `loading` 和 `disabled` 混在一起。前者是“正在执行”，后者是“不能执行”，这两个状态必须拆开。

---

## 三、插槽 / children 设计：给扩展留口子，但不放开结构

### 1. 推荐的扩展位

AI 输入框通常不是单一控件，而是一整块复合区。建议至少预留四类扩展位：

- **prefix**：前缀图标、模式标签
- **suffix**：快捷提示、状态说明
- **actions**：提交、取消、重试等操作区
- **helper**：辅助文案、字数提示、规范说明

```tsx
<AIInput
  prefix={<IconAI />}
  suffix={<ShortcutHint text="Enter 提交，Shift+Enter 换行" />}
  actions={
    <>
      <Button onClick={handleDraft}>润色一下</Button>
      <Button type="primary" onClick={handleSubmit}>生成</Button>
    </>
  }
  helper={<span>支持输入最多 500 字</span>}
/>
```

### 2. children 适合更复杂的组合

如果组件库更偏灵活，也可以做成 `children` 风格：

```tsx
<AIInput>
  <AIInput.Prefix>
    <Tag>标题优化</Tag>
  </AIInput.Prefix>

  <AIInput.Suffix>
    <span>建议包含品牌、规格、卖点</span>
  </AIInput.Suffix>

  <AIInput.Actions>
    <Button>插入模板</Button>
    <Button type="primary">生成优化建议</Button>
  </AIInput.Actions>

  <AIInput.Helper>
    输入越具体，生成结果越贴近业务目标
  </AIInput.Helper>
</AIInput>
```

这种写法结构清晰，适合复杂页面；但文档必须写明哪些子节点可选，以及在不同状态下哪些会被禁用或隐藏。

### 3. 两个业务扩展示例

**场景 A：客服话术生成**

```tsx
<AIInput
  prefix={<IconChat />}
  helper={<span>输入用户诉求，系统会生成回复草稿</span>}
  actions={
    <>
      <Button onClick={handleTemplate}>插入模板</Button>
      <Button type="primary" onClick={handleGenerate}>生成话术</Button>
    </>
  }
/>
```

**场景 B：商品标题优化**

```tsx
<AIInput
  prefix={<Tag>标题优化</Tag>}
  suffix={<span>建议包含品牌、规格、卖点</span>}
  actions={<Button type="primary">生成优化建议</Button>}
/>
```

同一个输入容器，靠插槽就能适配不同业务，而不会破坏统一视觉。

---

## 四、状态机设计：把空态、编辑态、提交态、禁用态、错误态拆开

### 1. 为什么必须做状态机

AI 输入框最容易出问题的不是布局，而是状态混乱。建议明确拆成五个状态：

- **空态**：没有输入内容
- **编辑态**：用户正在输入
- **提交态**：请求处理中
- **禁用态**：不可输入，不可提交
- **错误态**：提交失败，等待重试或修改

### 2. 状态流转表

| 当前状态 | 触发动作 | 下一个状态 | UI 表现 |
|---|---|---|---|
| 空态 | 输入文本 | 编辑态 | 显示占位文案和引导提示 |
| 编辑态 | 点击提交 / 回车 | 提交态 | 按钮 loading，输入区只读 |
| 提交态 | 请求成功 | 编辑态或空态 | 清空或保留输入，恢复操作 |
| 提交态 | 请求失败 | 错误态 | 显示错误文案和重试按钮 |
| 错误态 | 修改文本 | 编辑态 | 隐藏错误提示 |
| 错误态 | 点击重试 | 提交态 | 复用上次参数再次请求 |
| 任意状态 | 外部锁定 | 禁用态 | 全局灰化并禁止操作 |

### 3. 一个简单的伪代码示例

```ts
type AIInputState = 'empty' | 'editing' | 'submitting' | 'disabled' | 'error';

function getNextState(current: AIInputState, action: 'type' | 'submit' | 'success' | 'fail' | 'retry' | 'lock') {
  if (action === 'lock') return 'disabled';
  if (current === 'disabled') return 'disabled';

  switch (current) {
    case 'empty':
      return action === 'type' ? 'editing' : current;
    case 'editing':
      return action === 'submit' ? 'submitting' : current;
    case 'submitting':
      if (action === 'success') return 'editing';
      if (action === 'fail') return 'error';
      return current;
    case 'error':
      if (action === 'type') return 'editing';
      if (action === 'retry') return 'submitting';
      return current;
    default:
      return current;
  }
}
```

不要只靠一个 `loading` 包打天下。至少要把 `error` 和 `disabled` 单独区分开，因为：

- `loading` 是“正在执行”
- `disabled` 是“不能执行”
- `error` 是“执行失败后需要恢复”

状态拆清后，UI 反馈也会更清晰：空态展示引导文案，编辑态展示辅助提示，提交态展示进度和禁用按钮，错误态展示原因与重试入口，禁用态展示不可操作原因。

### 4. 按状态表现按钮与输入框

- **空态**：输入框可用，提交按钮默认禁用或低强调
- **编辑态**：输入框可编辑，提交按钮激活
- **提交态**：输入框只读，提交按钮 loading，取消按钮可见
- **禁用态**：输入框与所有按钮禁用
- **错误态**：输入框保持可编辑，重试按钮高亮

---

## 五、Hook 封装：让 UI 只负责展示，异步逻辑下沉

真正值得复用的，通常不是 UI 本身，而是输入、提交、重试、取消这一整套状态流转。可以把核心逻辑抽成 Hook：

```ts
const {
  value,
  state,
  setValue,
  submit,
  retry,
  cancel,
} = useAIInput({
  initialValue: '',
  onSubmit: async (payload) => {
    // 业务请求
  }
});
```

如果想进一步拆分，也可以把“状态管理”和“动作管理”分开：

```ts
const input = useAIInputState();
const action = useAIAction({ onSubmit, onRetry, onCancel });
```

推荐的职责划分是：

- `useAIInputState` 管受控值、状态流转、禁用判断
- `useAIAction` 管提交、重试、取消、错误恢复
- 组件只接收 Hook 暴露的状态和动作，不直接写请求细节

这样做的价值很直接：UI 更轻，业务更清晰，测试也更容易写。

---

## 六、可测试性、文档化与发布维护

### 1. 可测试性怎么做

AI 输入框至少要能测这些场景：

- 输入变化是否触发 `onChange`
- 回车提交是否触发 `onSubmit`
- loading 时按钮是否禁用
- error 时是否展示重试入口
- cancel 是否能回到编辑态
- disabled 时所有交互是否被阻断

建议把测试用例按状态来写，而不是按按钮来写。这样更接近真实交互，也更容易覆盖状态机。

### 2. 文档化模板

组件文档里最好固定写这些内容：

| 项目 | 示例 |
|---|---|
| 组件用途 | 收集 AI 输入并触发任务 |
| props | `value / placeholder / loading / disabled / mode / config` |
| 事件 | `onChange / onSubmit / onRetry / onCancel` |
| 插槽 | `prefix / suffix / actions / helper` |
| 状态 | `empty / editing / submitting / disabled / error` |
| 默认行为 | Enter 是否提交、空输入是否允许提交、失败后是否保留输入 |

文档里最好再补一段接入示例，让业务同学直接复制就能跑，不用翻源码理解默认规则。

### 3. 发布与维护

当这个组件进入公共库后，要约定好版本策略：

- 新增 `config` 项尽量保持向后兼容
- 事件 payload 扩展时，不破坏旧字段
- 重大交互变化要写变更说明
- 常用业务场景要补成示例仓库或 Storybook

如果后续要兼容“AI 建议面板”“AI 文本润色按钮”“AI 生成弹窗”“AI 结果卡片”，优先复用同一套状态机和 Hook，而不是复制一个新输入框出来。这样才能把 AI 能力真正沉淀为可复用资产。

---

## 结尾小结：把 AI 输入框做成真正的基础资产

AI 输入框看似只是一个输入组件，实际上是 AI 前端能力复用的起点。  
只要你把它设计成**清晰的 props、明确的事件、可扩展的插槽、稳定的状态机**，它就不再是某个页面的临时实现，而是可以在“AI 建议面板”“AI 文本润色按钮”“AI 生成弹窗”“AI 结果卡片”里反复复用的基础资产。

这一章最重要的三件事是：

1. **先定边界**：输入框只负责输入、提交、恢复，不吞业务结果。
2. **再定接口**：props、事件、插槽、config、状态机都要标准化。
3. **最后下沉逻辑**：把异步和重试收进 Hook，UI 只做展示。

如果你准备开始落地，可以直接按下面这个骨架推进：

- 先补齐 `value / placeholder / loading / disabled / mode / config`
- 再补 `onChange / onSubmit / onRetry / onCancel`
- 接着设计 `prefix / suffix / actions / helper`
- 然后把 `empty / editing / submitting / disabled / error` 画成状态表
- 最后把请求逻辑抽到 `useAIInput` 或 `useAIAction`

把这套结构做稳，后面的 AI 组件就有了统一的起点。

# 第4章 AI 建议面板：如何封装结果展示与交互选择

# AI 建议面板：如何封装结果展示与交互选择

在前端里，“建议”类 AI 能力最容易被写成一次性页面逻辑：输入内容、请求接口、渲染结果、用户点击结束。但一旦它出现在多个页面——表单补全、文案优化、商品标题推荐、模板选择——交互骨架几乎都是一样的：**展示区、选择区、操作区、空态、加载态**。  
所以，AI 建议面板不该是临时模块，而应该抽象成可复用的“结果展示 + 交互选择”组件，必要时再配一个 Hook 统一管理请求、状态拆分和事件分发。

---

## 一、为什么要把建议能力组件化

AI 建议面板的价值，不只是把接口结果展示出来，而是把一类高频交互变成稳定资产。组件化至少解决三件事：

1. **复用展示骨架**：标题、列表、操作区、空态、错误态直接复用  
2. **统一交互语义**：选择、接受、拒绝、刷新在不同页面保持一致  
3. **统一状态管理**：加载、禁用、无结果、部分可选按同一套规则处理

如果每个业务页都自己拼一套建议 UI，很快就会出现风格不一致、事件命名混乱、错误态缺失的问题。更合理的做法是：把建议面板做成通用组件，把请求和状态拆到 Hook 里，页面只接业务数据。

### 组件设计原则

建议遵循三个原则：

- **结构稳定，内容可变**：标题、列表、操作区固定，内容通过 props 注入
- **展示与决策分离**：面板只负责呈现与选择，不把提交逻辑写死
- **状态可见**：加载中、无结果、选择中、已接受都要有明确反馈

第一版不要把业务专属文案、接口返回结构、页面跳转逻辑塞进去，先做成通用容器。

---

## 二、通用交互抽象：先拆结构，再定职责

### 1. 面板结构拆解

一个通用的 AI 建议面板，至少包含四块：

- **标题区**：说明建议类型，必要时展示来源或生成状态
- **列表区**：展示建议项，支持单选或多选
- **操作区**：接受、拒绝、刷新、继续追问
- **空态区**：未触发、无结果、失败时的兜底展示

```text
┌──────────────────────────┐
│ 标题区 + 说明 + 状态标签   │
├──────────────────────────┤
│ 列表区：卡片 / 列表 / 侧栏 │
├──────────────────────────┤
│ 操作区：接受 / 拒绝 / 刷新 │
├──────────────────────────┤
│ 空态 / 错误态 / 占位态     │
└──────────────────────────┘
```

业务差异只影响“内容”，不影响“骨架”。表单补全、文案优化、模板选择都能落到同一套结构上。

---

## 三、组件 API 设计：props、事件与 children/renderers

### 1. 核心 props 设计

```ts
type SuggestionItem = {
  key: string
  title: string
  description?: string
  disabled?: boolean
  meta?: Record<string, any>
}

type Props = {
  items: SuggestionItem[]
  selectedKeys?: string[]
  loading?: boolean
  emptyText?: string
  variant?: 'card' | 'list' | 'aside'
  multiple?: boolean
}
```

### 2. 字段说明

| 字段 | 类型 | 含义 | 说明 |
|---|---|---|---|
| `items` | `SuggestionItem[]` | 建议列表 | 核心数据源 |
| `selectedKeys` | `string[]` | 当前选中项 | 受控模式下由外部维护 |
| `loading` | `boolean` | 加载状态 | 请求中禁用交互并展示占位 |
| `emptyText` | `string` | 空态文案 | 无数据或未触发时的提示 |
| `variant` | `'card' \| 'list' \| 'aside'` | 布局样式 | 也是布局策略 |
| `multiple` | `boolean` | 是否多选 | 默认单选，适合替换场景 |

`items` 至少包含 `key`、`title`，可选 `description`、`disabled`、`meta`。其中 `meta` 用来透传业务信息，比如原始输入、模板 ID、风格标签，避免把业务逻辑塞进组件内部。

`selectedKeys` 是受控字段，面板内部只负责展示选中态，最终结果由外部状态决定。单选常见于文案替换，多选常见于标签推荐、字段补全、模板组合。

`loading` 要控制请求中的禁用态，最好支持局部加载：骨架屏、按钮禁用、标题保留，而不是整块面板一刀切。`emptyText` 也建议区分“尚未触发”和“请求后无结果”。

`variant` 不只是样式参数，而是布局策略：卡片式、列表式、侧边栏式，对应不同页面空间和阅读习惯。

### 3. children / render props 扩展

为了让面板在不同业务里更灵活，建议保留插槽式扩展：

```tsx
<AiSuggestionPanel
  items={items}
  variant="card"
  headerExtra={<Tag>推荐</Tag>}
  footer={<button>查看全部</button>}
  renderItem={(item) => <SuggestionRow item={item} />}
/>
```

可扩展项建议包括：

- `renderItem(item)`：自定义每项渲染
- `headerExtra`：标题右侧附加区域
- `footer`：底部自定义操作区
- `emptyState`：外部接管空态渲染
- `errorState`：外部定制错误展示
- `maxHeight`：限制面板高度
- `disabledKeys`：统一控制禁用项

目标不是堆接口，而是让组件覆盖大多数场景，同时保留扩展面。

---

## 四、事件设计：onSelect、onAccept、onReject、onRefresh

### 1. 为什么事件要分层

AI 建议面板不是“点一下就结束”，而是一个决策流程，所以事件要按行为拆开，而不是合成一个 `onChange`。

### 2. 推荐事件定义

```ts
type Props = {
  onSelect?: (keys: string[], item: SuggestionItem) => void
  onAccept?: (keys: string[], items: SuggestionItem[]) => void
  onReject?: (keys: string[], items: SuggestionItem[]) => void
  onRefresh?: () => void
}
```

- `onSelect`：点选某项时触发，适合高亮预览、联动详情、实时更新右侧表单
- `onAccept`：确认采用时触发，这是写入业务状态的时机，比如替换输入框内容、回填字段
- `onReject`：明确不采用时触发，适合埋点、清理临时状态、切换下一轮建议
- `onRefresh`：重新生成时触发，请求逻辑建议交给上层 Hook 或容器管理

### 3. 事件时序

推荐闭环是：

1. 用户在 **AI 输入框** 输入上下文  
2. Hook 请求建议并填充面板  
3. 用户点击某一项触发 `onSelect`，只更新预览  
4. 用户点击“采用”触发 `onAccept`，同步到上游输入框或表单  
5. 用户点击“拒绝”触发 `onReject`，清空选择或进入下一轮  
6. 用户点击“刷新”触发 `onRefresh`，重新请求建议

### 4. 事件分工原则

- **选择**：改变当前焦点  
- **接受**：写入业务结果  
- **拒绝**：放弃当前结果  
- **刷新**：重新获取结果  

只要这四个动作定义清楚，组件职责就不会乱。尤其要避免“点击建议项就直接改表单”的耦合写法，否则后续加预览、对比、撤销会很难演进。

---

## 五、样式与状态规范：加载、禁用、视觉一致性

### 1. 不同状态要统一处理

建议面板常见状态包括：

- `loading`：请求中，按钮和列表要禁用
- `empty`：无数据或未触发
- `error`：请求失败，提供重试入口
- `partial`：部分建议不可用
- `selected`：当前选中项高亮

### 2. 状态机建议

```text
idle -> loading -> success
idle -> loading -> empty
idle -> loading -> error
success -> refreshing -> success/error
success -> selecting -> accepted/rejected
```

这能避免只写几个布尔值的混乱。比如 `loading`、`disabled`、`selectedKeys`、`error` 分开管理，逻辑会更清晰。

### 3. 视觉一致性

无论哪种 `variant`，都建议统一：

- 标题字号、间距、按钮位置
- 选中态颜色、禁用态样式
- loading、empty、error 的文案层级
- 图标和辅助信息的使用规则

用户一眼就能认出这是同一类 AI 能力，而不是不同团队拼出来的临时页面。视觉一致性本身就是可复用资产的一部分。

---

## 六、异步交互封装：把请求和状态拆进 Hook

建议面板最好配一个 Hook，例如：

```ts
const {
  items,
  selectedKeys,
  loading,
  error,
  refresh,
  selectItem,
  acceptItems,
  rejectItems,
} = useAISuggestions({ prompt, multiple: false })
```

这里的核心不是“把请求藏起来”，而是把状态拆分清楚：

- `items`：展示数据
- `selectedKeys`：选择状态
- `loading`：请求状态
- `error`：错误状态
- `refresh`：重新请求
- `selectItem/acceptItems/rejectItems`：事件分发

这样组件只管 UI，Hook 负责状态机，测试也更容易拆分：组件测渲染，Hook 测流转。

### Hook 封装

Hook 还适合统一处理：

- 请求防抖
- 重试逻辑
- 请求取消
- 最后一条结果覆盖前一条结果
- 与上游输入框联动的刷新时机

这能避免建议面板在不同页面里重复写“提交—等待—更新”的逻辑。

---

## 七、与输入框和表单联动：选中后如何回填

AI 建议面板通常不是孤立存在，它经常和 **AI 输入框**、表单字段、编辑器联动。

### 1. 卡片式用法

适合内容较长、需要比较的建议：

```tsx
<AiSuggestionPanel
  variant="card"
  items={items}
  selectedKeys={selectedKeys}
  loading={loading}
  emptyText="请先输入原始内容"
  onSelect={(keys, item) => setPreview(item)}
  onAccept={(keys, items) => {
    form.setFieldValue('title', items[0].title)
    form.validateField('title')
  }}
  onReject={() => setSelectedKeys([])}
  onRefresh={refreshSuggestions}
/>
```

这里的回填方式是：**面板不直接操作 DOM，而是回写上游状态**，再由表单接管展示和校验。

### 2. 列表式用法

适合条目短、数量多的建议：

```tsx
<AiSuggestionPanel
  variant="list"
  multiple
  items={items}
  selectedKeys={selectedKeys}
  loading={loading}
  emptyText="生成后可选择多个建议"
  onSelect={(keys) => setSelectedKeys(keys)}
  onAccept={(keys, items) => {
    setTags(items.map((i) => i.title))
  }}
  onReject={() => setSelectedKeys([])}
  onRefresh={refreshSuggestions}
/>
```

同一套 API 可以同时支撑单选与多选，也能支撑卡片式和列表式布局。页面只需要根据业务决定 `variant`、`multiple` 和回填策略。

### 3. 继续操作

如果用户点击“继续生成”，建议把当前 `selectedKeys`、原始输入、`meta` 一起传给下一轮请求，这样新结果能继承上下文，而不是每次从零开始。比如“继续追问”可以把已选建议作为上下文再次提交，让生成链路连续推进。

---

## 八、错误态和兜底策略：别让面板只剩成功路径

### 1. 错误态要可恢复

当请求失败时，面板不要直接空白。至少要给出：

- 错误提示
- 重试按钮
- 是否保留上一次结果

如果保留旧结果，建议标识“当前结果已过期”，避免用户误以为是最新数据。

### 2. 兜底策略

- 接口超时：显示重试，保留输入上下文
- 返回空数组：显示无结果态，提供重新生成入口
- 部分项不可用：保留展示并置灰
- 生成失败：保留操作区，允许继续追问或切换输入

这类兜底在真实业务里很重要，因为 AI 建议功能往往是“高频 + 低容错”的交互。

---

## 九、文档化、可测试性与发布维护

### 1. 文档模板

文档至少包含：

- 适用场景
- `props` 表格
- 事件说明
- `variant` 差异
- `loading / empty / error` 处理方式
- 基础用法、受控用法、插槽扩展示例

建议 README 结构如下：

```md
## 用途
## Props
## Events
## Slots / Renderers
## 状态说明
## 示例
## 常见问题
```

### 2. 可测试性

重点测试：

- `items` 为空时是否正确显示空态
- `loading` 时按钮和列表是否禁用
- `selectedKeys` 更新后样式是否同步
- `onSelect/onAccept/onReject/onRefresh` 是否触发正确
- 不同 `variant` 下布局是否一致
- `disabled` 项是否不可选但仍可展示原因

如果有 Hook，还要单测请求成功、失败、重试、刷新、取消等状态流转。组件测 UI，Hook 测状态机，这是最稳妥的拆法。

### 3. 发布与维护

发布前建议检查：

- 是否保持 props 向后兼容
- 是否补齐文档和示例
- 是否为新 `variant` 增加回归测试
- 是否有默认空态与错误态
- 是否会影响现有页面的视觉一致性

版本升级时尽量采用渐进式变更：先新增能力，再弃用旧字段，不要一次性打断所有接入方。维护时最好保持统一命名：`items`、`selectedKeys`、`loading`、`emptyText`、`variant`、`onSelect`、`onAccept`、`onReject`、`onRefresh`，这样后续在 **AI 输入框、AI 建议面板、AI 文本润色按钮、AI 生成弹窗、AI 结果卡片** 之间切换时，学习成本最低。

---

## 小结：把“建议”做成资产，而不是页面临时逻辑

AI 建议面板的核心，不是“能展示几个结果”，而是能否形成稳定的前端资产：**统一结构、统一事件、统一状态、统一布局策略**。  
当你把它封装成组件或 Hook 后，同样的能力就能复用到输入框建议、表单修正、内容润色、模板推荐、结果确认等多个页面。

这一章最重要的三件事是：

1. 先定骨架，再定数据和事件  
2. 用 `props + events + slots/renderers + Hook` 拆清职责  
3. 把空态、加载态、错误态、部分可选态一起纳入设计

直接可抄的落地顺序如下：

1. 先定骨架：标题、列表、操作区、空态  
2. 再定 props：`items`、`selectedKeys`、`loading`、`emptyText`、`variant`  
3. 再定事件：`onSelect`、`onAccept`、`onReject`、`onRefresh`  
4. 再补状态：加载、禁用、错误、空态  
5. 最后写文档和测试，保证能在多个业务页稳定复用

只要这条链路打通，AI 建议面板就不再是某个页面的附属功能，而会成为组件库里一个可维护、可扩展、可复用的能力模块。

# 第5章 AI 文本润色按钮：从单点触发到标准操作组件

# AI 文本润色按钮：从单点触发到标准操作组件

## 开篇引入：为什么“润色”不该只是一个临时按钮

很多项目里的 AI 文本润色，一开始都很临时：页面放一个“润色一下”按钮，点击后把当前文本发给接口，结果直接覆盖原文。能跑通，但一旦接入表单、富文本编辑器、详情页、评论框，问题就会暴露：

- 同样能力要复制多套逻辑；
- 不同页面对加载态、失败态、权限态、空内容态的要求不一致；
- 回填规则变化时，比如从“替换全文”改成“只替换选区”，原实现往往要重写。

所以这一章不是教你做一个“能用”的按钮，而是把它抽象成一个**标准操作组件**：既能作为轻量按钮单独使用，也能挂接到输入区、编辑器选区、表单字段里，成为可复用的前端资产。目标不是一个按钮，而是一套能力。

---

## 一、AI 文本润色按钮适合哪些场景？

### 1. 适合“短文本、局部编辑、即时反馈”的场景

AI 文本润色按钮最适合处理“用户已经写了一部分内容，需要快速优化”的情况，而不是从零生成大段内容。典型场景包括：

- 标题、备注、简介优化；
- 富文本里对选中文本改写；
- 评论、回复、客服话术语气调整；
- CMS 或运营后台的一句话文案优化；
- 将口语表达改成更正式、更简洁的文本。

这些场景的共同点是：**输入已存在，动作明确，结果可回填**。用户要的是快，而不是多轮对话。

### 2. 不适合强行按钮化的场景

如果任务需要多轮追问、复杂条件判断、长文本结构化重写，单个按钮就不合适，更适合抽屉、弹窗、侧边栏或生成面板。判断标准很简单：

- 1 次点击能否完成主动作？
- 结果是否能直接回填当前位置？
- 用户是否更在意“快”而不是“过程”？

大多是“是”，就适合做成按钮；否则不要硬塞进按钮里。

---

## 二、按钮 API 怎么设计：把 props 先定清楚

### 1. 核心 props：text、prompt、scope、loading、disabled

可复用的 AI 润色按钮，至少要把“输入内容、润色指令、作用范围、状态控制”抽出来，而不是把逻辑写死在组件内部。

```tsx
<AiPolishButton
  text={value}
  prompt="请将这段话润色为更专业、简洁的表达"
  scope="selection"
  loading={polishing}
  disabled={!value}
/>
```

#### `text`
待润色文本，建议做成**受控数据**，由宿主页面掌握最终文本，组件只负责触发能力，不私自维护容易失真的副本。

#### `prompt`
润色指令不要写死。同一个组件在不同业务里可能要做：

- 更正式
- 更简洁
- 更适合客服回复
- 更适合营销文案
- 更适合详情页描述

把 `prompt` 做成可配置项，组件才真正可复用。

#### `scope`
表示润色作用范围，建议至少抽象为：

- `input`：普通输入框全文
- `selection`：选中文本
- `field`：表单字段
- `block`：富文本块级内容

`scope` 的价值在于让按钮知道自己处理哪一段内容，而不是只会发请求。

#### `loading` 与 `disabled`
这两个状态要分开：

- `loading`：请求中，禁止重复点击；
- `disabled`：当前不允许操作，比如文本为空、权限不足、字段未聚焦、正在保存。

很多组件把二者混成一个 `disabled`，结果“不可点”和“处理中”语义混乱，样式和文案也难统一。

### 2. 推荐的默认值和约束

为了让接入方少踩坑，建议给出明确默认值：

| prop | 默认值 | 说明 |
| --- | --- | --- |
| `text` | `""` | 空字符串时一般不可执行 |
| `prompt` | `""` | 允许业务显式传入，不建议写死 |
| `scope` | `"input"` | 默认按输入框全文处理 |
| `loading` | `false` | 由组件内部或宿主状态控制 |
| `disabled` | `false` | 外部可额外禁用权限或空态 |

同时建议补一个约束：`text` 为空且 `scope` 不是特殊模式时，按钮自动进入不可用态，避免无效请求。

## 三、事件怎么设计：让按钮的状态流转可追踪

### 1. 事件设计：onStart、onSuccess、onError、onComplete

工程化按钮不能只返回结果，还要把过程拆开，方便埋点、回填、提示和收尾。

```tsx
<AiPolishButton
  text={value}
  prompt="请润色为更简洁的表达"
  onStart={({ text, scope }) => setPolishing(true)}
  onSuccess={({ result }) => setValue(result)}
  onError={(err) => message.error(err.message)}
  onComplete={() => setPolishing(false)}
/>
```

#### 事件流转图

```text
idle
  └─点击触发
      ├─onStart
      ├─loading
      ├─请求成功 ──> onSuccess ──> success
      ├─请求失败 ──> onError   ──> error
      └─finally   ──> onComplete ─> idle/ready
```

### 2. 什么时候能再次触发

建议把“是否能再次点击”说死，避免业务各写各的：

- 处于 `loading` 时不可再次触发；
- `onComplete` 执行完后恢复可点击；
- 若文本在请求中被用户修改，应以新内容重新计算可用态；
- 如果 `scope="selection"` 且当前已无选区，按钮应回到禁用态。

换句话说，**按钮是否可再次触发，取决于 loading 是否结束、输入是否仍有效、作用范围是否仍存在**。

### 3. `onStart / onSuccess / onError / onComplete` 各自负责什么

- `onStart`：锁定界面、清理旧错误、记录埋点；
- `onSuccess`：只负责回写结果，不做复杂副作用；
- `onError`：展示错误、给出重试入口、保留原文；
- `onComplete`：统一收尾，关闭 loading，恢复交互。

把职责切开后，组件就更像标准操作，而不是一次性事件回调。

## 四、children 和 icon：同一逻辑，三种视觉形态

### 1. 让按钮既能标准化，也能风格化

同一个 AI 润色能力，可能出现在工具栏、输入框右侧、表格操作列、卡片底部，因此按钮必须允许不同外观。建议同时支持 `children` 和 `icon`。

```tsx
<AiPolishButton icon={<SparkIcon />} text={value} />
<AiPolishButton>一键润色</AiPolishButton>
<AiPolishButton icon={<SparkIcon />}>AI 优化</AiPolishButton>
```

### 2. 三种常见形态

#### 图标按钮
适合编辑器工具栏或紧凑布局。此时 `icon` 负责表达能力，通常需要配合 tooltip，例如“AI 润色”。

#### 文字按钮
适合表单字段、列表操作区。`children` 直接承载文案，像“AI 优化”“润色表达”“重新措辞”。

#### 图标 + 文字按钮
适合既要占位明确、又要降低学习成本的场景。这个形态最适合首次上线，用户一眼就知道按钮做什么。

### 3. 保持同一交互逻辑，不同外观只是皮肤变化

无论是图标按钮、文字按钮还是带下拉的菜单项，内部执行流程都应该一致：

1. 校验 `text` 是否为空；
2. 进入 `loading`；
3. 触发 `onStart`；
4. 发起润色请求；
5. 成功后触发 `onSuccess`；
6. 失败后触发 `onError`；
7. 最后触发 `onComplete`。

这样做的好处是：视觉可以变，交互契约不变，后续接入和测试都更稳定。

### 4. 视觉一致性要统一在哪些点

外观可以变，但下面这些规则建议统一：

- loading 动画和按钮尺寸；
- disabled 透明度和不可点击样式；
- 错误态颜色；
- hover、active 反馈；
- 图标与文字间距；
- 成功后提示样式。

否则同一个能力在不同页面看起来像不同产品，维护成本会很高。

## 五、如何挂接到输入区、编辑器和表单字段？

### 1. 文本输入区：最简单的接法

输入框场景通常是全文润色，核心是把输入值传给按钮，再把结果回写回去。

```tsx
<AiPolishButton
  text={text}
  prompt="请润色为更简洁的表达"
  scope="input"
  onSuccess={({ result }) => setText(result)}
/>
```

注意两点：

- 文本为空时应直接 `disabled`；
- 不要自动触发，必须显式点击，避免用户输入中途被打断。

### 2. 编辑器选区：按钮要知道“润色哪一段”

富文本场景里，按钮通常处理的是当前选区，而不是全文。此时 `scope="selection"` 很关键。按钮不需要知道编辑器细节，但必须把“当前选区文本”和“替换结果”这两个边界约定清楚，宿主才能正确回填。

```tsx
<AiPolishButton
  scope="selection"
  text={editor.getSelectionText()}
  prompt="请将选中的内容改写得更自然"
  icon={<SparkIcon />}
/>
```

如果编辑器支持插槽，也可以把按钮作为工具栏 slot 的一部分注入进去，保持编辑器主体和 AI 能力解耦。

### 3. 表单字段：把按钮做成字段附属能力

在表单中，AI 润色按钮通常不是主角，而是字段右侧的增强操作。推荐这样包裹：

```tsx
<FormField label="商品描述">
  <Textarea value={value} onChange={setValue} />
  <AiPolishButton
    text={value}
    prompt="请改写为更适合商品详情页的描述"
    onSuccess={({ result }) => setValue(result)}
  >
    AI 优化
  </AiPolishButton>
</FormField>
```

关键点是：**组件只提供能力，字段状态仍由表单系统管理**。这样校验、提交、重置才能保持一致，不会出现 AI 按钮和表单状态脱节的问题。

## 六、错误处理与兜底：按钮不能只管成功

### 1. 防重复提交：把按钮当成有状态操作，而不是普通点击

AI 润色通常是异步请求，用户连续点击很容易发出多次请求。推荐三层保护：

- **前端立即锁定**：点击后立刻切换到 `loading`，禁止再次触发；
- **请求幂等标识**：同一文本、同一 scope 的连续请求可用 `requestId` 区分；
- **结果只认最后一次**：如果用户在请求期间又修改了文本，旧结果不要覆盖新内容。

简单说，就是“谁最后确认，谁才写回”。

### 2. 失败回退：原文永远是兜底答案

AI 按钮最容易被忽略的是失败后的体验。建议至少准备三层兜底：

1. **请求失败**：展示可重试提示，不要吞掉错误；
2. **结果为空**：保留原文，不要强行覆盖；
3. **用户取消或超时**：恢复按钮状态，避免界面卡死。

另外要坚持一个原则：**失败不污染原文**。无论润色接口失败多少次，原文本都必须能回退。对输入框、表单和编辑器来说，这一点非常关键，也是用户敢不敢用的前提。

### 3. 错误态的交互建议

错误态不要只给一条 toast，最好同时提供：

- 错误文案；
- 重试按钮；
- 原文预览；
- 可选的恢复到上一次成功结果。

这样按钮才真正像一个可维护的标准操作组件，而不是一个“点一下试试”的临时入口。

## 七、组件文档：别让接入方靠猜

一个可复用的 AI 按钮，文档至少要写清楚这些内容：

- 适用场景：输入框、选区、表单字段；
- props：`text`、`prompt`、`scope`、`loading`、`disabled`；
- 事件：`onStart`、`onSuccess`、`onError`、`onComplete`；
- `children` / `icon` 用法；
- 状态：默认态、加载态、错误态、禁用态；
- 回填方式：替换全文、替换选区、回写字段；
- 失败兜底：如何保留原文、如何重试；
- 可测试性：哪些状态要写单测，哪些交互要做回归。

文档建议再补一个最小模板，直接让业务方照着接：

```md
### Props
| name | type | default | description |
| --- | --- | --- | --- |
| text | string | "" | 待润色文本 |
| prompt | string | "" | 润色指令 |
| scope | 'input' \| 'selection' \| 'field' \| 'block' | 'input' | 作用范围 |
| loading | boolean | false | 请求中状态 |
| disabled | boolean | false | 不可操作状态 |

### Events
- onStart(payload)
- onSuccess(payload)
- onError(error)
- onComplete(payload)
```

文档越清楚，业务方接入时越少反复确认“这个按钮到底怎么接”。

## 八、发布与维护：怎样让按钮成为可长期复用的资产？

### 1. 先抽 Hook，再包组件

工程上建议拆成两层：

- `useAiPolish`：负责请求、状态机、错误处理、回调；
- `AiPolishButton`：负责 UI 展示、交互入口、视觉一致性。

这样不仅按钮能用，Hook 也能复用于菜单项、快捷键、右键菜单、弹窗操作区。

```tsx
const { run, loading, error } = useAiPolish({
  prompt,
  onSuccess,
  onError,
});
```

Hook 封装的好处是：业务如果不是按钮，而是“菜单项触发润色”或“表单自动建议”，也能复用同一套状态逻辑。

### 2. 组件设计原则要统一

这个按钮后续要长期维护，建议坚持几条原则：

- **职责单一**：只负责触发润色，不在组件里写业务回填规则；
- **状态拆分**：`loading`、`disabled`、`error`、`success` 分开管理；
- **配置优先**：通过 `prompt`、`scope`、`children`、`icon` 适配变化；
- **视觉一致性**：不同业务里保持统一尺寸、间距、禁用样式；
- **可测试性**：关键状态都能通过单测覆盖。

### 3. 版本迭代要守住接口稳定

一旦这个按钮被多个业务页面复用，props 和事件就是公共契约。后续新增能力时，优先做可选项扩展，不要随意改旧语义。比如可以增加：

- `retryable`
- `maxLength`
- `transformResult`
- `beforeRequest`

这类扩展比直接改原字段更稳，也更符合组件库的发布维护方式。

## 结尾小结：把一个按钮做成可复用能力

AI 文本润色按钮看起来只是一个小操作，但它很适合练习“组件化 AI 能力”的基本功：把输入、指令、作用范围、状态、事件、样式、回填方式拆清楚，再通过 Hook 封装和组件包装落到多个业务页面。

你可以按下面清单落地：

- 是否把 `text`、`prompt`、`scope` 设计成可配置项；
- 是否区分了 `loading` 和 `disabled`；
- 是否拆分了 `onStart / onSuccess / onError / onComplete`；
- 是否支持 `children` 和 `icon` 扩展；
- 是否明确了输入区、选区、表单字段的接入方式；
- 是否有重复提交保护、错误回退、可测试性和文档化；
- 是否能用 `useAiPolish` 复用到按钮之外的交互入口。

当一个按钮可以稳定复用到多个页面时，它就不再是一次性的交互，而是团队可以持续积累的 AI 前端资产。

# 第6章 AI 生成弹窗：复杂任务的交互容器设计

# AI 生成弹窗：复杂任务的交互容器设计

## 开篇引入：什么时候该用“生成弹窗”

当 AI 功能从“点一下出结果”变成“先配置、再执行、再确认”时，单个按钮或简单面板就不够了，这时更适合用 **AI 生成弹窗**。

它的价值不在于“多一个弹层”，而在于把一段复杂任务收口成一条清晰流程：

1. 用户进入弹窗；
2. 填写必要配置；
3. 触发 AI 执行；
4. 预览生成结果；
5. 确认应用或回退修改。

常见场景包括：

- 根据表单内容生成文案；
- 根据当前页面数据生成配置建议；
- 先输入条件，再生成结构化结果；
- 生成后需要用户确认是否写回页面。

如果硬塞进普通按钮或小浮层，容易出现流程断裂、状态混乱、结果难确认。生成弹窗的意义，就是把 **配置、执行、预览、确认** 四段能力统一管理，并沉淀成可复用组件。

---

## 一、AI 生成弹窗适合承载什么复杂流程？

### 1.1 先判断：它是不是“需要上下文的 AI 任务”

适合弹窗的任务通常满足以下任一条件：

- 需要用户先提供多个参数；
- 生成结果不是最终答案，而是待确认草稿；
- 执行过程较慢，需要独立展示加载态；
- 结果出来后要支持预览、编辑、再次生成；
- 需要把生成内容应用回当前页面某个区域。

判断标准很简单：**只要它是一段有前后关系的流程，就适合用弹窗来承载。**

### 1.2 不适合的情况

如果只是一个单点动作，比如“AI 文本润色按钮”点一下立刻替换选中文本，更适合按钮 + 下拉面板，而不是完整弹窗。

但如果这个按钮背后还有：

- 选择语气、长度、受众；
- 先生成多个版本；
- 预览后再应用；

那它就应该升级为生成弹窗入口。判断依据不是“有没有 AI”，而是“有没有需要确认的流程”。

### 1.3 设计原则

组件设计原则是：**把复杂流程收口到单一容器，但不要把业务逻辑写死在容器里。**  
弹窗负责流程组织，生成内容、字段结构、校验规则都应通过 `props`、`children` 或配置注入，这样才能复用到内容编辑页、运营配置页、表单详情页等不同场景。

---

## 二、弹窗结构：配置区、执行区、预览区、确认区怎么拆？

### 2.1 配置区：收集生成所需输入

配置区负责承载用户输入，常见内容包括：

- 主题词、目标受众、语气；
- 生成长度、模板类型；
- 关联上下文数据；
- 额外开关项，如“是否包含标题”。

建议把配置区当成独立输入模块，而不是弹窗里的临时表单。可以通过 `children` 注入表单，也可以用 `steps` 配置驱动多步骤流程。

```tsx
<AiGenerateModal
  open={open}
  title="生成活动文案"
  initialValues={{ tone: '正式', length: '中等' }}
>
  {(ctx) => (
    <GenerateForm
      values={ctx.values}
      onChange={ctx.setValues}
    />
  )}
</AiGenerateModal>
```

这里 `children` 适合高度定制化表单；如果多个页面复用同类字段，则可把表单再抽成子组件。弹窗本体只关心：当前是否可编辑、数据是否完整、是否可以提交。

### 2.2 执行区：明确展示“正在生成”

执行区的重点不是好看，而是让用户知道系统正在做什么。常见元素有：

- loading spinner；
- 进度提示；
- 当前步骤说明；
- 禁用重新提交按钮。

如果 AI 调用需要几秒甚至更久，用户最怕的是“点了没反应”。因此执行区的文案要具体，比如“正在分析输入内容”“正在生成建议”“正在整理结果”。加载态不仅是视觉反馈，更是交互约束：告诉用户哪些操作可用，哪些不可用。

### 2.3 预览区：让结果先“可见再可用”

预览区是生成弹窗的核心。它不只是展示文本，而是让用户判断：

- 结果是否符合预期；
- 是否需要再次生成；
- 是否允许局部编辑；
- 是否可以直接应用到页面。

如果是 `AI 结果卡片`，可以把结果拆成标题、摘要、标签、正文等结构化字段，方便后续复用。  
如果是 `AI 建议面板`，则可以展示多条候选建议，并支持单条采纳。

建议把预览区设计成可替换渲染区域：

```tsx
<AiGenerateModal
  renderPreview={(result) => <ResultCard data={result} />}
/>
```

这样在不同业务中，预览可以是纯文本、卡片、表格或列表，避免组件内部绑死一种展示方式。

### 2.4 确认区：把“看过”变成“真正应用”

确认区是很多 AI 组件容易漏掉的一环。  
预览不等于确认，确认必须提供明确动作：

- “应用到页面”；
- “替换当前内容”；
- “插入到编辑器”；
- “保存并关闭”。

确认按钮和提交按钮最好分开命名，避免用户误解：提交是触发 AI，确认是应用结果。尤其在生成弹窗里，按钮文案一定要和状态匹配：生成阶段用“开始生成”，预览阶段用“应用结果”。

## 三、props 设计：让弹窗像标准组件一样可接入

### 3.1 基础 props 设计

一个可复用的 `AI 生成弹窗`，核心 props 可以这样设计：

```ts
type StepStatus = 'idle' | 'editing' | 'submitting' | 'previewing' | 'confirming' | 'error';

type AiGenerateModalProps = {
  open: boolean;
  title?: string;
  steps?: StepConfig[];
  initialValues?: Record<string, any>;
  submitText?: string;
  confirmText?: string;
  loading?: boolean;
  children?: React.ReactNode | ((ctx: ModalContext) => React.ReactNode);
  renderPreview?: (result: any) => React.ReactNode;
};
```

#### `open`
控制弹窗显隐，建议由外部统一管理，避免内部自己打开/关闭导致业务不同步。

#### `title`
标题应支持动态配置，比如：

- “生成营销文案”
- “生成页面建议”
- “生成字段描述”

#### `steps`
用于多步骤流程配置。每一步可以包含：

- step 标识；
- 标题；
- 校验规则；
- 渲染内容；
- 是否需要等待 AI 返回。

#### `initialValues`
用于预填默认值，尤其适合从当前页面上下文带入初始内容。

#### `submitText`
按钮文案应可配置，避免所有场景都写“提交”。例如：

- “开始生成”
- “生成建议”
- “立即润色”

#### `confirmText`
确认阶段的文案，建议明确表达“应用”而不是“提交”。

#### `loading`
允许外部接管加载态，便于和全局请求状态、页面禁用逻辑统一。

#### `children`
用于承载配置区。它既可以是静态节点，也可以是 render function，把当前 `values`、`setValues`、`status` 注入进去。

#### `renderPreview`
用于自定义预览区，适合 `AI 结果卡片`、`AI 建议面板`、列表、表格等不同展示形态。

### 3.2 一套更完整的参数表

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| `open` | boolean | false | 是否展示弹窗 |
| `title` | string | - | 弹窗标题 |
| `steps` | StepConfig[] | [] | 多步骤配置，未传时按单步处理 |
| `initialValues` | object | {} | 表单初始值 |
| `submitText` | string | “开始生成” | 执行按钮文案 |
| `confirmText` | string | “应用结果” | 确认按钮文案 |
| `loading` | boolean | false | 外部加载状态 |
| `children` | ReactNode / render function | - | 自定义配置区内容 |
| `renderPreview` | function | - | 自定义结果预览区 |
| `onOpenChange` | function | - | 打开/关闭变化回调 |
| `onSubmit` | function | - | 触发生成 |
| `onConfirm` | function | - | 确认应用结果 |
| `onClose` | function | - | 关闭后收尾 |

### 3.3 让 props 服务于复用

设计 props 时要遵循一个判断标准：**只要是多个业务页面都可能变化的内容，就不要写死在组件内部。**

比如：

- 弹窗标题可能变化；
- 步骤数可能变化；
- 默认值可能变化；
- 提交文案可能变化；
- 结果展示区可能变化。

而组件内部应该固定流程骨架：打开、编辑、生成、预览、确认。这样组件才有公共能力的价值。

## 四、事件设计：把流程拆成可测试的动作

### 4.1 四个关键事件

```ts
type AiGenerateModalEvents = {
  onOpenChange?: (open: boolean) => void;
  onSubmit?: (values: Record<string, any>) => Promise<any>;
  onConfirm?: (result: any) => Promise<void> | void;
  onClose?: () => void;
};
```

#### `onOpenChange`
负责弹窗开合的统一出口。关闭时要同步清理临时状态，避免下一次打开还残留旧结果。

#### `onSubmit`
负责触发 AI 生成，建议返回 Promise，方便组件内部自动处理 loading、成功、失败状态。回传参数通常是当前配置区的 `values`，必要时也可附带 `step`、`context`。

#### `onConfirm`
负责应用生成结果。这个动作通常发生在预览之后，适合和 `onSubmit` 解耦。回传参数通常是当前 `result`，如果允许局部修改，也应带上最终编辑后的结果对象。

#### `onClose`
负责关闭后的收尾，例如重置步骤、清空错误信息、埋点记录等。

### 4.2 一条完整交互链路

以“商品标题优化”为例：

1. 打开弹窗，`onOpenChange(true)`；
2. 在配置区填写商品类目、关键词、语气，值进入 `initialValues` 管理的表单状态；
3. 点击“开始生成”，触发 `onSubmit(values)`；
4. 进入 `submitting / generating` 状态，提交按钮禁用，配置区按需禁用；
5. 生成成功后进入预览态，展示 `ResultCard`；
6. 用户点击“应用结果”，触发 `onConfirm(result)`；
7. 确认成功后关闭弹窗，触发 `onClose()` 并清理临时结果。

### 4.3 事件设计的误区

常见错误是把所有逻辑都塞进 `onSubmit`，结果会变成：

- 生成和确认混在一起；
- 无法支持预览后再次编辑；
- 错误回退难做；
- 测试也难拆分。

正确做法是把“提交生成”和“确认应用”拆成两个动作，中间通过状态机串起来。这样每个事件职责单一，Hook 也更容易封装。

## 五、多步骤状态机：怎么处理加载、禁用、回退？

### 5.1 为什么要用状态机思路

生成弹窗本质上是一个小流程系统，不是单一表单。  
如果没有状态机，组件很快会变成一堆 `if else`：

- 打开时能不能点提交？
- 生成中能不能改配置？
- 预览中能不能再次提交？
- 失败后是重试还是回到编辑？

建议把状态拆清楚：

```txt
idle -> editing -> submitting -> previewing -> confirming -> confirmed
                    \-> error -> editing
```

### 5.2 状态机与 UI 变化

| 状态 | 配置区 | 提交按钮 | 确认按钮 | 预览区 | 说明 |
|---|---|---|---|---|---|
| `idle` | 关闭 | - | - | - | 弹窗未打开 |
| `editing` | 可编辑 | 可点击 | 禁用 | 空态 | 正在填写参数 |
| `submitting` | 禁用或只读 | loading | 禁用 | 空态 | 正在发起生成 |
| `previewing` | 可选只读 | 可重新生成 | 可点击 | 展示结果 | 用户检查结果 |
| `confirming` | 只读 | 禁用 | loading | 展示结果 | 正在应用结果 |
| `error` | 可编辑 | 可重试 | 禁用 | 错误提示 | 失败后回退 |
| `confirmed` | 关闭或清空 | - | - | - | 已应用结果 |

### 5.3 加载态和禁用态

在 `submitting` 或 `confirming` 阶段：

- 提交按钮禁用；
- 配置区可按需禁用；
- 关闭按钮是否可用要按业务决定；
- 需要明确显示当前状态文案。

如果允许用户在生成过程中继续编辑，就必须定义“编辑后是否自动重试”或“是否需要重新提交”，否则状态会漂移。

### 5.4 回退、取消和重试

回退不是简单返回上一步，而是要回答：**回退后保留什么、清掉什么、重新请求什么。**

常见策略：

- 回退到编辑态时保留表单值，不保留生成结果；
- 失败后保留错误提示和输入内容；
- 再次提交前清空旧结果，避免误判为新生成；
- 如果预览支持局部修改，则修改后应标记为“未确认”。

取消也要区分两种：

- **关闭取消**：直接退出弹窗，通常保留上次输入的草稿；
- **流程取消**：明确放弃当前生成结果，清掉临时状态。

重试按钮建议只在 `error` 或 `previewing` 下出现，避免多个请求并发。

### 5.5 状态拆分建议

建议至少拆成三层状态：

1. **可见状态**：open / closed；
2. **流程状态**：editing / submitting / previewing / error / confirming；
3. **数据状态**：values / result / errorMessage。

这样写 Hook 时也能更清晰地暴露能力，形成可复用资产，而不是把所有状态揉成一个大对象。

## 六、Hook 封装：把流程状态从页面里抽出去

AI 生成弹窗不要只做成 UI，还要有一层 Hook 封装状态，方便多个页面复用。

```ts
function useAiGenerateModal() {
  const [open, setOpen] = useState(false);
  const [step, setStep] = useState<'editing' | 'submitting' | 'previewing' | 'error' | 'confirming'>('editing');
  const [values, setValues] = useState({});
  const [result, setResult] = useState(null);
  const [errorMessage, setErrorMessage] = useState('');

  return {
    open,
    step,
    values,
    result,
    errorMessage,
    setOpen,
    setValues,
    setResult,
    setErrorMessage,
    reset: () => {
      setStep('editing');
      setValues({});
      setResult(null);
      setErrorMessage('');
    }
  };
}
```

Hook 封装的好处是：

- 页面只关心业务数据；
- 弹窗只关心流程展示；
- 状态复用更稳定；
- 单测也更容易针对 Hook 断言。

## 七、一个完整业务案例：生成结果预览后再确认提交

假设你在商品运营页里做“标题优化”。

### 第一步：配置区

用户输入：

- 商品类目：保温杯；
- 关键词：大容量、便携；
- 语气：简洁专业；
- 输出数量：3 条。

这些内容通过 `initialValues` 填入，用户在 `children` 渲染的表单里修改。

### 第二步：执行区

点击“开始生成”后，弹窗进入加载态：

- `submitText` 按钮变成 loading；
- 配置区禁用；
- 底部提示“正在生成标题建议”。

### 第三步：预览区

生成完成后，不直接写回页面，而是展示 `AI 结果卡片`：

```tsx
const renderPreview = (result) => (
  <AiResultCard
    title={result.title}
    tags={result.tags}
    content={result.content}
    actions={[
      { label: '重新生成', type: 'secondary' },
      { label: '应用结果', type: 'primary' }
    ]}
  />
);
```

这一步的关键是：用户先看，再决定要不要用。  
如果结果不满意，点“重新生成”回到编辑态；如果满意，点“应用结果”触发 `onConfirm(result)`。

### 第四步：确认区

确认时可以把结果写回当前页面，或者仅保存到草稿区。  
若确认失败，应保留当前预览结果，并提示失败原因，避免用户白看一轮。

这个案例体现了弹窗的价值：**不是生成本身，而是把生成后的决策流程也纳入组件能力。**

## 八、组件文档示例：让别的页面接得上

一个 AI 生成弹窗能否真正复用，最终取决于文档是否足够清楚。文档至少要写明：

- 组件用途：适合什么场景；
- props 表：每个字段的类型、默认值、是否必填；
- 事件表：什么时候触发、参数是什么；
- 状态说明：每个状态下 UI 长什么样；
- 示例代码：至少提供一个完整可运行示例。

### 示例说明

```tsx
<AiGenerateModal
  open={open}
  title="生成结果摘要"
  submitText="开始生成"
  confirmText="应用结果"
  initialValues={{ source: '当前表单' }}
  onOpenChange={setOpen}
  onSubmit={handleGenerate}
  onConfirm={handleApply}
  onClose={handleClose}
  renderPreview={renderPreview}
/>
```

文档里还应补充：

- `onSubmit` 失败时如何展示错误；
- `steps` 不传时是否默认单步；
- `children` 与 `steps` 是否可以同时使用；
- 预览区是否允许自定义渲染；
- `loading` 由内部控制还是外部接管。

为了降低接入成本，建议再附一段状态说明：

- `editing`：填写参数；
- `submitting`：等待返回；
- `previewing`：检查结果；
- `error`：可重试；
- `confirming`：正在应用；
- `confirmed`：已应用结果。

这些内容直接关系到可测试性和维护成本。

## 九、发布与维护：别让组件“能用但难养”

### 9.1 发布前检查清单

- props 是否有默认值；
- 事件是否都可追踪；
- loading 与 disabled 是否一致；
- 错误态是否可重试；
- 关闭弹窗是否会正确重置状态；
- 文档是否包含完整示例；
- 是否有基础单测覆盖状态切换。

### 9.2 可测试性建议

测试重点不要放在“AI 返回什么内容”，而要放在：

- 打开后是否进入正确状态；
- 提交时按钮是否禁用；
- 失败后是否回到编辑态；
- 确认后是否调用 `onConfirm`；
- 关闭后是否清理状态；
- 重试后是否重新进入 `submitting`。

如果状态拆分做好，测试就会从“测页面”变成“测流程”，稳定很多。

### 9.3 维护原则

后续维护时，优先保证：

- 结构稳定，业务差异通过配置扩展；
- 状态命名一致；
- 按钮文案和流程语义清晰；
- 预览区和确认区职责分明；
- 新增场景尽量复用现有 Hook，而不是复制一份弹窗；
- 视觉一致性保持统一，避免同类 AI 组件出现不同的间距、按钮层级和禁用态样式。

---

## 结尾小结：把复杂 AI 流程收进一个可复用容器

AI 生成弹窗的本质，不是“弹窗里放了一个 AI”，而是把一段复杂任务抽象成可复用的交互容器。你要做的不是写一个一次性页面，而是沉淀出一套能在多个业务页面复用的前端资产。

记住这几个落地要点：

- 用弹窗承载“配置 + 执行 + 预览 + 确认”完整流程；
- 用 `props` 控制差异，用 `events` 拆分动作；
- 用状态机管理加载、禁用、回退；
- 用 `children` 或 `steps` 承载可变内容；
- 用 Hook 封装状态，用组件承载视觉与交互；
- 用文档、测试、默认值保证可维护性。

当你能把 AI 生成弹窗做成标准能力，后面的 AI 输入框、AI 建议面板、AI 文本润色按钮、AI 结果卡片，都会变成同一套体系下的不同形态。

# 第7章 AI 结果卡片：让输出可视化、可操作、可复用

# AI 结果卡片：让输出可视化、可操作、可复用

AI 接入前端后，最容易失控的不是“生成”，而是“展示”。如果每个页面都把接口返回值直接塞进 `div`，很快就会出现三个问题：

1. 展示风格不统一；
2. 复制、重试、接受、继续编辑等操作重复实现；
3. 结果结构一变，页面代码就跟着改，复用性很差。

所以，**AI 结果不要只当字符串渲染，而要包装成标准结果卡片**。它不是单纯的展示组件，而是一个可复用的交互容器：承接 AI 输出、维护展示状态、暴露用户动作，适合放在列表、详情页、弹窗、侧边栏等场景复用。

## 一、结果卡片的典型结构：标题、摘要、正文、操作区

一个合格的 AI 结果卡片，通常是分层组织的。

### 1）标题区：告诉用户“这是什么结果”

标题至少要传达两件事：

- 业务名称，例如“AI 文案建议”“AI 审批意见”
- 结果状态，例如“已生成”“待确认”“生成失败”

标题尽量短，保证识别性，例如：

- `AI 文案建议`
- `商品标题优化结果`
- `会议纪要摘要`

必要时可加轻量标签，如“可直接应用”“需要人工确认”，让用户快速判断当前阶段。

### 2）摘要区：先给结论，再给细节

摘要区放最重要的一句话，帮助用户决定要不要继续看正文，例如：

- “建议将标题前置核心卖点，并减少形容词。”
- “当前结果包含 3 条修改建议，适合直接应用到编辑器。”
- “生成已完成，但有 1 处字段缺失需要补充。”

摘要的目标不是完整表达，而是**降低阅读成本**。信息较长时，它也可以作为折叠前的预览。

### 3）正文区：展示结构化或半结构化内容

正文区承载主要输出，常见形式包括：

- 一段纯文本
- 分点建议
- 表格化字段
- 富文本片段
- JSON 风格的结构化渲染

关键原则是：**正文适配结果类型，而不是强行统一样式**。文本按段落渲染，列表按列表渲染，字段用 key-value 结构渲染，才能保证可读性和可操作性。

### 4）操作区：让结果可执行

操作区是结果卡片的价值放大器，常见操作包括：

- 复制结果
- 接受结果
- 重试生成
- 继续编辑
- 替换原文
- 查看详情

操作区最好固定在卡片底部，形成稳定预期。用户不需要每次重新找按钮，也不用猜这张卡片能不能改、能不能拿走。

---

## 二、组件设计原则：先抽象通用交互，再保留业务扩展

结果卡片不要一开始就绑定某个页面，而要先抽出通用交互：

- 结果展示
- 状态切换
- 操作触发
- 复制与重试兜底
- 页面级扩展区

核心原则是：**把稳定部分做成组件，把变化部分留给 props 和 children**。如果某个能力会反复出现在 AI 输入框、AI 建议面板、AI 文本润色按钮、AI 生成弹窗里，它就应该抽象成公共能力，而不是散落在页面逻辑中。

---

## 三、props 设计：result、status、actions、variant、copyable 怎么拆

要想复用，关键在于 **props 足够抽象，但不抽象过头**。建议把输入拆成“内容”“状态”“行为”“外观”四类。

### 1）`result`：结果内容本身

`result` 是核心数据源，可以是字符串，也可以是对象。推荐统一成业务适配后的模型，而不是直接把后端原始结构扔进组件。

```ts
type ResultCardData = {
  title: string
  summary?: string
  content: string | Array<{ label: string; value: string }>
  meta?: Record<string, string>
  sourceText?: string
}
```

如果业务里结果类型很多，建议上层先适配成统一模型，不要让组件直接依赖后端结构。这样组件更稳定，后端字段变动时只需要改适配层。

### 2）`status`：控制状态机

`status` 建议明确枚举，而不是用一堆布尔值拼起来。常见状态包括：

- `idle`
- `loading`
- `success`
- `error`
- `disabled`

内部展示逻辑会更清晰：`loading` 显示骨架屏，`error` 显示错误文案和重试按钮，`disabled` 禁用交互。

如果需要更细的语义，可以拆成两层，例如“请求状态”和“结果状态”。这就是**状态拆分**：不要把所有语义压到一个字段里，避免后续扩展时状态爆炸。

### 3）`actions`：定义允许哪些操作

`actions` 可以用配置数组控制底部按钮：

```ts
actions: ['copy', 'accept', 'retry', 'edit']
```

也可以更细：

```ts
actions: [
  { key: 'copy', label: '复制' },
  { key: 'accept', label: '接受' },
  { key: 'retry', label: '重试' },
  { key: 'edit', label: '继续编辑' },
]
```

同一个卡片就能在不同页面显示不同操作，比如详情页只保留“复制、接受”，弹窗里只保留“重试、继续编辑”。

### 4）`variant`：控制外观语义

`variant` 用来表达展示风格，而不是业务逻辑，例如：

- `default`
- `compact`
- `highlight`
- `inline`

`compact` 适合列表页，`highlight` 适合确认页，`inline` 适合编辑器旁边的小区域。它也可以作为视觉一致性的统一入口，避免每个页面各写一套样式。

### 5）`copyable`：控制是否支持复制

复制能力不要写死，应该显式配置：

- `copyable: true`
- 或 `copyable: { format: 'text' | 'plain', successMessage?: string }`

这样可以控制只复制纯文本，避免富文本样式一起带走；也方便统一复制成功文案，例如“已复制建议内容”。

---

## 四、事件设计：onCopy、onAccept、onRetry、onEdit 如何统一

事件设计的关键是：**卡片只负责发出意图，不负责业务决策**。

### 1）`onCopy`

用户点击复制时触发。事件参数最好带上当前结果和复制文本，方便外层决定复制全文、摘要还是某个字段，也方便埋点统计和复制成功提示。

```ts
onCopy?: (payload: {
  result: ResultCardData
  text: string
  type: 'summary' | 'content' | 'all'
}) => void
```

如果复制失败，组件不要吞掉错误，至少给出兜底提示，例如“复制失败，请手动选中后复制”，并保留按钮可再次点击。

### 2）`onAccept`

“接受”通常表示把结果应用到业务上下文里，例如写回表单、替换编辑器内容、更新列表字段。卡片本身不直接改状态，只通过 `onAccept(result)` 通知外层处理，这样组件就不会被具体业务绑死。

### 3）`onRetry`

重试一般和异步状态绑定，常见于生成失败或用户不满意时。卡片内部只提供按钮，真正的重试逻辑放到 Hook 或容器层，避免组件变成“半个请求模块”。

这里可以结合 **Hook 封装**：比如 `useAiResultCard()` 统一处理重试、加载、复制、状态回写，卡片只接收 `status` 和回调，页面不必重复写请求流转。

```ts
const { status, retry, copy } = useAiResultCard({
  resultId,
  onSuccess: setResult,
  onError: showToast,
})
```

如果重试需要携带参数，也可以把上一次请求上下文一起传出去。失败后建议保持原结果不清空，只在卡片顶部提示“重试中”或“重试失败，可再次点击”，这样用户不会丢上下文。

### 4）`onEdit`

“继续编辑”通常意味着跳回编辑态，适合和弹窗、抽屉、编辑器联动，例如在详情页点击后打开编辑器，或在卡片下方展开可修改区域。这个事件最重要的是命名统一、语义清晰，不要一会儿叫 `onModify`，一会儿叫 `onChangeResult`。

---

## 五、不同结果类型的展示策略：文本、结构化数据、混合内容

AI 输出不总是长文本，结果卡片要能按内容类型切换渲染策略。

### 1）文本型结果

适合直接显示段落、列表、建议句。建议支持换行、关键点高亮，并控制最大高度，避免卡片无限拉长。文本较长时，最好提供“展开/收起”，让摘要和全文分层显示。

这类场景很适合放在 `AI 文本润色按钮` 之后，用户点击润色后直接看到前后对比与结果内容。

### 2）结构化数据结果

例如字段建议、标签推荐、步骤拆解，更适合表格或 key-value 布局：

- 字段名：标题
- 建议值：提高核心词权重
- 说明：减少空泛修饰语

这种布局可读性更高，也更容易被“接受”按钮直接应用。需要逐条确认时，结构化展示也方便局部勾选、局部复制和局部替换。

### 3）混合内容结果

最常见的是“摘要 + 列表 + 备注”。建议按层级拆分：

- 先展示摘要
- 再展示结构化项
- 最后展示补充说明或注意事项

不要把所有内容揉成一坨文本。混合场景里，卡片最好支持 `children` 或插槽，让上层插入“风险提示”“来源说明”“对比前后文”等额外模块。

```tsx
<ResultCard
  result={data}
  status="success"
  actions={['copy', 'accept', 'edit']}
>
  <ResultCard.Footer>
    <span>基于当前商品描述生成</span>
  </ResultCard.Footer>
</ResultCard>
```

这种插槽设计很重要，因为不同业务页面经常需要补一行上下文，而不是重新造一个新组件。`children` 也可以用于插入二次确认区、来源标记、版本号等信息。

---

## 六、状态机、加载与禁用态：让卡片在异步场景里稳定工作

AI 结果卡片一定要和状态机一起设计，否则加载中、失败后、已接受后的表现会混乱。

### 1）推荐的状态拆分

可以按三层拆：

- 请求状态：`loading / success / error`
- 交互状态：`disabled / active`
- 结果状态：`pending / confirmed / applied`

这样卡片在不同业务里更容易复用，不会把所有状态揉成一个大对象。比如结果已生成但当前页面不允许编辑，可以是 `success + disabled`，而不是直接隐藏组件。

### 2）加载态

加载态不要只转圈，最好给出占位信息，例如：

- “正在生成建议…”
- 骨架屏占位
- 预估等待提示

如果请求耗时较长，尽量保留标题区和空白正文区，让布局稳定，减少页面抖动。

### 3）禁用态

当上游未准备好、权限不足或结果已锁定时，卡片应进入禁用态：

- 按钮不可点
- 提示原因
- 保留结果展示但降低交互噪音

禁用态不是简单变灰，而是要明确告诉用户为什么不能操作，比如“当前草稿未保存，暂不可应用结果”。

### 4）错误态与兜底

错误态至少要回答三个问题：

- 发生了什么
- 用户现在能做什么
- 是否可以重试

常见兜底文案如：

- “结果生成失败，请稍后重试”
- “当前内容不完整，无法直接应用”
- “仅支持文本复制，其他格式已降级展示”

如果错误来自格式不符合预期，建议做降级渲染，至少保住可读信息，不要直接空白。

---

## 七、三种复用场景：列表页、详情页、弹窗内怎么放

结果卡片不是只给一个页面用的，它应该成为跨页面能力块。

### 1）列表页：压缩展示

在列表中，卡片应偏 `compact`：

- 只显示标题、摘要、少量正文
- 按钮减少到 1-2 个
- 避免过高占位影响列表密度

列表页更重视“扫一眼就知道是什么”，所以视觉重心要放在标题、状态和一个主操作上。

### 2）详情页：完整展示

在详情页中，卡片可以展示完整正文和全部操作：

- 适合查看完整生成结果
- 适合放“接受”“继续编辑”“复制”三类动作
- 适合结合历史版本

这时可以用 `children` 承载补充信息，例如来源字段、生成时间、版本记录，让卡片既能展示结果，也能承接业务上下文。

### 3）弹窗：强调确认

在弹窗里，卡片的职责通常是确认结果是否可用：

- 先看摘要
- 再决定接受或重试
- 操作区和弹窗按钮保持一致

这类场景下，`variant` 和 `actions` 配置尤其重要。若弹窗只是确认，就不要把完整编辑功能都塞进去，否则会破坏聚焦。

---

## 八、`AI 生成弹窗`、`AI 输入框`、`AI 建议面板` 如何串起来

结果卡片不是孤立组件，它通常是整条 AI 交互链路的收口。

一个常见组合是：

- `AI 输入框`：收集用户原始输入
- `AI 建议面板`：展示可选建议或提示词
- `AI 生成弹窗`：承接生成前确认和参数配置
- `AI 结果卡片`：承接生成后的可视化和后续操作

这条链路里，结果卡片负责把输出变成可操作资产，而不是只展示一段文本。比如用户在弹窗里点了“生成”，卡片先显示 `loading`，生成完成后切成 `success`，然后提供“复制”“接受”“继续编辑”三个动作。这种状态机过渡，才是组件化的价值。

---

## 九、文档化、可测试性与发布维护

一个组件能不能复用，不只看抽象是否合理，还看文档和测试是否跟上。

### 文档至少要写清楚

- `result` 的数据格式
- `status` 各状态的视觉表现
- `actions` 支持哪些操作
- `onCopy / onAccept / onRetry / onEdit` 的触发时机
- 不同 `variant` 的适用场景
- 复制、禁用、错误态的示例
- `children` / 插槽允许放什么内容

建议附一个 props 表格，直接让团队照着接：

| props | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `result` | `ResultCardData` | 是 | 结果内容 |
| `status` | `'idle' \| 'loading' \| 'success' \| 'error' \| 'disabled'` | 是 | 当前状态 |
| `actions` | `ActionKey[]` | 否 | 显示哪些操作 |
| `variant` | `'default' \| 'compact' \| 'highlight' \| 'inline'` | 否 | 外观变体 |
| `copyable` | `boolean \| object` | 否 | 是否允许复制 |
| `children` | `ReactNode` | 否 | 扩展内容 |

### 测试至少覆盖

- 空数据渲染
- loading 状态
- error 状态与重试
- 复制成功回调
- 接受/编辑事件触发
- 不同结果类型的分支渲染
- `actions` 为空时的默认行为
- `disabled` 状态下按钮不可点击

如果结果卡片没有测试，后续每接一个业务都等于多一次回归风险。尤其当它成为公共组件后，任何小改动都可能影响多个页面，所以可测试性不是加分项，而是必要项。

发布时也要把 README、示例页和变更记录一起带上，避免团队只会“拿来用”，却不知道它支持哪些状态、哪些插槽、哪些事件。

---

## 小结：把 AI 输出从“结果文本”升级成“可复用能力”

AI 结果卡片的核心，不是“把内容框起来”，而是把输出变成标准化前端资产：**可展示、可操作、可配置、可复用**。

实现时，记住这几个原则：

- 用 `result + status + actions + variant + copyable` 拆分职责
- 用 `onCopy / onAccept / onRetry / onEdit` 表达用户意图
- 用状态机管理加载、禁用、错误与确认态
- 按文本、结构化数据、混合内容三类结果分别渲染
- 通过 `children` 或插槽扩展额外信息，避免组件封死
- 在列表、详情页、弹窗中通过不同配置复用同一个组件
- 把“复制后继续编辑再提交”作为标准链路设计进去
- 配套文档、测试、示例页和发布说明，避免“只有一个页面能用”

如果你把 AI 结果卡片设计好，后面的 AI 输入框、AI 建议面板、AI 文本润色按钮、AI 生成弹窗，都会顺着同一套工程规范往下长。

# 第8章 组件 API 设计原则：把 AI 能力做成稳定接口

# 组件 API 设计原则：把 AI 能力做成稳定接口

## 开篇引入：先把“能用”变成“可复用”

AI 能力进到前端组件库后，最常见的问题不是做不出来，而是只能在某一个页面临时跑通：今天加一个润色按钮，明天加一个生成弹窗，后天又在表单里塞一个建议面板。如果每次都按页面临时拼装，最后只会留下大量难维护、难复用的代码。

组件库里的 AI 功能，不能只是一次性功能块，而要被抽象成稳定接口，像普通 UI 组件一样可理解、可组合、可替换。这样业务页面拿来就能用，接入成本低，后续维护也不会被某个页面的特殊写法拖乱。

核心目标只有一个：**把一次性的 AI 交互，做成可复用的前端资产**。而实现这件事的关键，就是 API 设计。

---

## 一、单一职责与最小能力集：这个组件到底只该负责什么？

### 1. 一个组件只解决一种交互任务

AI 组件最容易失控的地方，是把输入、请求、结果展示、错误提示、埋点、权限判断全塞进一个组件里。看起来是一体化，实际上是把不同层次的职责揉成一团，后续一改就会牵一发而动全身。

更稳妥的方式，是按交互职责拆分：

- **AI 输入框**：收集输入、触发提交、展示加载与禁用态。
- **AI 建议面板**：展示建议列表，支持选择、忽略、刷新。
- **AI 文本润色按钮**：对当前文本发起润色动作，结果交给外部处理。
- **AI 生成弹窗**：承载生成表单、结果确认、取消与重试。
- **AI 结果卡片**：负责结果渲染、复制、采纳、重新生成等展示动作。

这样拆完，语义清晰，业务方一看名字就知道该放在哪里，不需要读实现去猜它到底能干什么。

### 2. 最小能力集怎么判断

判断标准很简单：**这个组件离开某个能力后，是否还能自然存在**。

比如“AI 文本润色按钮”本质是动作触发器，它的职责是发起润色，不该顺手接管整套编辑器逻辑；“AI 建议面板”本质是结果容器，不该顺手承担输入和请求流程。最小能力集越清晰，组件越容易复用，也越容易测试。

### 3. 不该继续新增 props 的信号

当你开始频繁补字段时，要先停下来判断是不是设计失控了。下面这些情况，通常不该继续往组件里加 props，而应考虑拆分组件或下沉到 Hook：

- 新增字段只服务某一个页面。
- 一个参数只在极少数分支里有意义。
- 新旧字段重叠，例如 `showLoading`、`loadingText`、`loadingNode` 同时出现。
- 组件已经需要区分“商品标题优化”“商品描述优化”“问答生成”三套规则。

这时继续加 props，只会把 API 推向“万能组件”。正确做法是把差异抽出去，保持接口收敛。

---

## 二、统一 API 设计清单：先定规则，再定字段

做 AI 组件库，不是每个组件都凭感觉起字段名，而是要先统一一套设计清单。

### 1. 单一职责

先写一句话定义组件用途：

- AI 输入框：负责输入和提交。
- AI 建议面板：负责展示和选择。
- AI 文本润色按钮：负责触发润色动作。
- AI 生成弹窗：负责承载生成流程。
- AI 结果卡片：负责展示结果与后续操作。

如果一句话说不清，就说明职责还没收敛。

### 2. 受控 / 非受控

先判断状态归谁管：

- 需要和表单联动时，用受控：`value`、`open`、`selectedItems`。
- 只做局部交互时，用非受控：`defaultValue`、`defaultOpen`。

例如 AI 生成弹窗如果外部已经在管理页面流程，就传 `open` 和 `onOpenChange`；如果只是一个临时弹层，就允许内部自己维护开关状态。

### 3. 默认值策略

默认值要安全、可预期，不能把业务偏好写死：

- `placeholder` 只负责提示输入，不替代业务文案。
- `loadingText` 提供统一加载反馈。
- `mode` 默认选最常见行为，如 `'replace'` 或 `'insert'`。
- `disabled` 默认 `false`，避免组件一挂载就不可用。

### 4. 版本兼容策略

字段命名一旦定下来，就尽量稳定：

- 避免同义字段并存，比如 `showLoading` 和 `loading` 不要同时出现。
- 新能力优先新增可选项，不要改旧语义。
- 删除字段前先给替代方案和迁移期。

这样能显著降低组件库升级时的接入成本。

---

## 三、配置项、事件、插槽 / children：API 怎么拆才不乱？

### 1. 配置项负责“长相和策略”

配置项适合表达静态能力和行为策略，例如：

- `placeholder`
- `maxLength`
- `autoFocus`
- `showRetry`
- `loadingText`
- `emptyText`
- `mode: 'insert' | 'replace' | 'append'`

这些参数描述的是“组件默认怎么表现”，适合放在 props 里。比如 AI 输入框可以通过 `placeholder` 提示“请输入需要润色的内容”，通过 `maxLength` 控制边界，通过 `loadingText` 给出加载反馈。

### 2. 事件负责“状态变化”

事件只做一件事：把变化通知外部。常见包括：

- `onSubmit(inputValue)`
- `onGenerate(params)`
- `onSuccess(result)`
- `onError(error)`
- `onRetry()`
- `onAccept(payload)`
- `onDismiss(item)`

命名尽量统一，用动词开头，让开发者一眼知道发生了什么。比如 AI 文本润色按钮点击后，组件内部只负责触发 `onGenerate`，请求、结果落地、编辑器替换都交给外层页面或 Hook 处理。

### 3. 插槽 / children 负责“内容可扩展”

当组件需要支持业务定制时，不要把所有文案和结构硬编码进去，而是留出插槽或 `children`：

- `header`：自定义 AI 建议面板标题区
- `footer`：自定义按钮区
- `empty`：无结果时的空状态
- `error`：错误态展示
- `children`：自定义结果卡片主体

例如 AI 结果卡片可以设计成：

- `title`
- `description`
- `actions`
- `children`

默认情况下，组件提供基础布局；业务方如果要展示“对比前后文案”或“只渲染部分结果”，就可以通过 `children` 接管主体，而不用重写整个组件。

### 4. 一套统一的 API 模板

为了避免不同 AI 组件各写各的，建议统一成下面这种结构：

```ts
type AIComponentProps = {
  value?: string
  defaultValue?: string
  loading?: boolean
  disabled?: boolean
  placeholder?: string
  maxLength?: number
  mode?: 'insert' | 'replace' | 'append'

  onChange?: (value: string) => void
  onSubmit?: (value: string) => void
  onSuccess?: (result: unknown) => void
  onError?: (error: Error) => void

  header?: React.ReactNode
  footer?: React.ReactNode
  empty?: React.ReactNode
  error?: React.ReactNode
  children?: React.ReactNode

  status?: 'idle' | 'loading' | 'success' | 'error' | 'disabled'
}
```

这套模板的价值不在于字段多，而在于结构稳定：业务方知道去哪里找配置、去哪里挂事件、去哪里自定义内容、去哪里看状态。

---

## 四、错误的 API 设计 vs 推荐设计

### 1. 错误示例：把业务逻辑写死在组件里

```tsx
<AIRewriteButton
  articleType="product"
  productId="123"
  useEditorReplace={true}
  autoSubmit
  showRetryText
  showProductHint
/>
```

问题很明显：

- `articleType`、`productId` 是业务字段，组件不该认识。
- `useEditorReplace`、`autoSubmit`、`showProductHint` 混在一起，职责不清。
- 后续换成“问答生成”或“标题优化”时，这套字段就失效了。

### 2. 推荐示例：组件只管通用交互，业务放外层

```tsx
<AIRewriteButton
  loading={rewriteState.loading}
  disabled={!canRewrite}
  mode="replace"
  onGenerate={handleRewrite}
  onSuccess={handleRewriteSuccess}
  onError={handleRewriteError}
>
  生成润色结果
</AIRewriteButton>
```

这里组件只暴露通用能力：

- `loading` 控制状态
- `disabled` 控制可用性
- `mode` 控制结果处理方式
- `onGenerate`、`onSuccess`、`onError` 控制流程

业务字段交给页面层或 Hook 处理，组件就能复用到不同场景。

### 3. 再看 AI 建议面板和结果卡片

```tsx
<AISuggestPanel
  items={suggestions}
  loading={loading}
  empty={<EmptyState />}
  onSelect={handleSelect}
  onDismiss={handleDismiss}
  footer={<RefreshButton onClick={refresh} />}
/>
```

```tsx
<AIResultCard
  title="AI 生成结果"
  description="点击采纳后写入编辑器"
  actions={[
    { key: 'accept', label: '采纳', onClick: handleAccept },
    { key: 'copy', label: '复制', onClick: handleCopy },
  ]}
>
  {content}
</AIResultCard>
```

这种设计的好处是，props 负责参数，事件负责变化，children 负责定制，结构天然适合复用。

---

## 五、状态拆分、加载与禁用态：别让 UI 和请求互相打架

### 1. 至少拆四类状态

建议至少拆成四类状态：

- **输入状态**：用户正在输入什么
- **请求状态**：是否加载、是否禁用、是否可重试
- **结果状态**：结果内容、选择项、采纳状态
- **错误状态**：错误文案、错误码、兜底动作

状态拆得越清楚，组件越容易测试，也越不容易出现“加载中还能点提交”的问题。

### 2. 加载与禁用态要明确

加载态不是简单转个 spinner。比如：

- AI 输入框在请求中应禁用提交按钮。
- AI 建议面板加载时只禁用刷新，不一定禁用查看旧结果。
- AI 文本润色按钮在执行中应进入 `loading`，并阻止重复点击。
- AI 生成弹窗在提交后可禁用表单，但保留取消或关闭能力。

建议把 `loading` 和 `disabled` 区分开：`loading` 代表正在处理，`disabled` 代表当前不可操作。二者语义不同，别混成一个布尔值。

### 3. Hook 封装承接状态机

如果多个组件共享同一种请求流程，可以抽成 Hook：

- `useAIGenerate(options)`
- `useAIRewrite(options)`
- `useAISuggest(options)`

Hook 负责：

- 调用接口
- 处理状态机：`idle / loading / success / error`
- 暴露 `run / retry / reset`
- 返回统一格式结果

组件只负责渲染和交互。这样“能力”和“皮肤”就分开了，页面可以自由组合。

---

## 六、命名规范与默认值：让接口一眼能读懂

### 1. 命名先统一，再扩展

推荐保持同类字段一致：

- 展示类：`placeholder`、`emptyText`、`errorText`
- 状态类：`loading`、`disabled`、`status`
- 行为类：`onChange`、`onSubmit`、`onRetry`、`onAccept`

不要在同一套组件里混用 `submitText` 和 `confirmText`、`open` 和 `visible`、`onClose` 和 `onDismiss`。命名不统一，文档和测试也会一起混乱。

### 2. 默认值要站在“安全接入”一边

默认值策略要避免误伤业务：

- `disabled` 默认 `false`
- `loading` 默认 `false`
- `maxLength` 不默认过小
- `mode` 默认只选一种最常见行为
- `children` 缺省时提供完整基础布局

这样即使业务方只传最少参数，组件也能稳定工作。

---

## 七、检查清单：一个 AI 组件 API 是否合格？

在发布前，可以快速检查：

- 是否职责单一，能用一句话说清用途？
- 是否同时支持受控与非受控，但规则明确？
- 配置项、事件、插槽 / children 是否分层合理？
- 是否有清晰的加载与禁用态？
- 错误态和兜底策略是否可配置？
- 命名是否统一，默认值是否安全？
- 是否避免写死业务逻辑？
- 是否提供可测试的状态入口和事件出口？
- 文档是否写清楚 props、事件、插槽、状态机？

如果这些问题有一项说不清，就不要急着发布。组件库里的 AI 能力，最怕的不是“少一个功能”，而是“接口不稳定”。

---

## 结尾小结：把 AI 组件做成资产，而不是一次性页面代码

AI 前端能力真正能沉淀下来，靠的不是“功能多”，而是**接口稳**。稳定的组件 API，应该让业务方一看就懂：怎么传参数、怎么监听事件、怎么自定义内容、怎么处理加载和错误、怎么和外部状态联动。

记住这条路径：

1. 先做**单一职责**
2. 再定**受控 / 非受控**
3. 接着分好**配置项、事件、插槽**
4. 统一**命名、默认值、版本策略**
5. 最后把业务差异下沉到 **Hook** 和外部页面

当你把 AI 输入框、AI 建议面板、AI 文本润色按钮、AI 生成弹窗、AI 结果卡片都按这套原则设计时，它们就不再是临时功能，而是真正可以在多个业务页面复用的前端资产。

**行动建议：**
- 给现有 AI 组件补一份统一 API 表。
- 检查是否存在“一个组件管太多”的情况。
- 把请求流程迁到 Hook，组件只保留交互和展示。
- 为每个组件补齐 `loading`、`disabled`、`error`、`retry` 四类状态说明。
- 统一命名和默认值，避免同义字段并存。
- 把业务专属逻辑从组件里移出，留给页面层或 Hook。

# 第9章 样式、状态与视觉一致性：让 AI 组件融入体系

# 样式、状态与视觉一致性：让 AI 组件融入体系

## 开篇引入：为什么 AI 组件最容易“看起来不属于这里”

很多团队把 AI 功能做出来后，最先暴露的问题不是“能不能用”，而是“像不像系统里的东西”。

同样是 `AI输入框`，在 A 页面里像主流程的一部分，到了 B 页面却像临时塞进来的工具条。`AI文本润色按钮`、`AI建议面板`、`AI生成弹窗`、`AI结果卡片` 也常常各画各的，最后把整个系统的视觉节奏打乱。

根因通常不是 AI 逻辑，而是**样式体系、状态体系和视觉规范没有组件化**。  
如果要把 AI 能力做成可复用资产，就不能只封装交互，还要把**视觉语言**一起抽出来：谁是主入口，谁是辅助操作，加载、错误、禁用怎么表现，尺寸和密度怎么适配，都应在组件层统一表达。AI 组件要同时满足两件事：**能工作**，以及**看起来属于这个设计系统**。

---

## 一、样式体系与设计 token 的接入方式：AI 组件怎么和设计系统对齐？

### 1.1 先接 token，不要先写死颜色和间距

AI 组件最常见的失控点，就是直接写死 `#1677ff`、`12px`、`8px` 这类值。短期省事，长期会让组件无法适配主题、暗色模式和不同业务品牌，尤其是跨页面复用时，后面只能不断打补丁。

建议把样式来源分成三层：

- **基础 token**：颜色、字体、圆角、间距、阴影
- **组件 token**：输入框高度、按钮尺寸、建议面板间距、结果卡片背景
- **业务配置**：是否紧凑、是否强调、是否显示图标、是否显示辅助文案

例如，`AI输入框` 不要直接依赖固定色值，而应使用 `colorPrimary`、`colorBorder`、`radiusMd`、`spacingSm` 这类 token。

```tsx
<AIInput
  value={value}
  onChange={setValue}
  token={{
    colorBorder: 'var(--color-border)',
    colorPrimary: 'var(--color-primary)',
    radiusMd: 8,
    spacingSm: 8,
    inputHeight: 40,
  }}
/>
```

### 1.2 组件样式要支持“可配置但不失控”

样式最好通过少量、明确的 props 暴露，而不是把所有 CSS 决策都交给页面：

- `size="sm | md | lg"`
- `variant="default | subtle | emphasis"`
- `density="comfortable | compact"`
- `theme="light | dark | auto"`
- `className` / `style` 作为最后兜底

原则是：**token 管默认，props 管差异，className 管特例**。  
比如 `AI文本润色按钮` 默认走系统主色，但某个页面想更低调，可以传 `variant="subtle"`；编辑器工具栏空间紧张，可以传 `size="sm"`。但不要把 `padding`、`borderRadius`、`boxShadow` 全开放给业务，否则视觉一致性很快失控。

```tsx
<AIPolishButton
  size="sm"
  variant="subtle"
  onClick={handlePolish}
  disabled={!selectedText}
>
  文本润色
</AIPolishButton>
```

### 1.3 用组件 token 统一 AI 能力的外观语言

建议进一步抽出组件 token，例如：

- `aiInputHeight`
- `aiPanelGap`
- `aiCardRadius`
- `aiErrorColor`
- `aiLoadingColor`

这样 `AI建议面板` 和 `AI结果卡片` 就不会各自定义一套“差不多但不一样”的样式。统一 token 的直接收益是：改主题只改一处，改密度也只改一处，维护成本明显下降。

```ts
const aiTokens = {
  aiInputHeight: 40,
  aiPanelGap: 12,
  aiCardRadius: 10,
  aiErrorColor: 'var(--color-error)',
  aiLoadingColor: 'var(--color-primary)',
};
```

---

## 二、加载态、禁用态、空态、错误态、成功态：AI 组件的状态怎么统一？

### 2.1 先统一状态规范，再讨论视觉

AI 组件很容易出现“状态各说各话”。同样是请求中，有的页面显示骨架屏，有的显示转圈，有的按钮变灰但没有提示。用户看到的不是同一个系统，而是几个临时拼起来的局部方案。

建议至少统一五类状态：

- **加载态**：正在请求 AI 能力
- **禁用态**：条件不满足，用户不能触发
- **空态**：没有输入、没有结果、没有历史建议
- **错误态**：请求失败、超时、服务不可用、输入不合法
- **成功态**：请求完成，结果可用，可复制、可替换、可插入

### 2.2 用表格把视觉表现和交互限制说清楚

| 状态 | 视觉表现 | 交互限制 | 适用组件 |
| --- | --- | --- | --- |
| 加载态 | loading icon、骨架屏、按钮文案变为“处理中” | 禁止重复提交，部分操作锁定 | AI输入框、AI建议面板、AI生成弹窗 |
| 禁用态 | 置灰、降低对比度、提示原因 | 不可点击，不触发请求 | AI文本润色按钮、AI结果卡片 |
| 空态 | 引导文案、占位图、轻提示 | 不展示结果操作 | AI建议面板、AI结果卡片 |
| 错误态 | 错误色、重试按钮、局部提示 | 保留可恢复操作 | AI输入框、AI生成弹窗、AI建议面板 |
| 成功态 | 正常主色、结果高亮、操作可用 | 可复制、可替换、可插入 | AI结果卡片、AI生成弹窗 |

### 2.3 每种状态都要有明确的字段命名

不要把这些状态写成一堆布尔值。比如 `isLoading && isDisabled && hasError` 会让组件不可维护。更清晰的方式是明确字段：

- `requestStatus: 'idle' | 'loading' | 'success' | 'error'`
- `uiMode: 'editable' | 'readonly' | 'preview'`
- `dataState: 'empty' | 'partial' | 'filled'`
- `interactionState: 'normal' | 'hover' | 'active' | 'disabled'`

这样 `AI结果卡片` 的边框、按钮、提示语、footer 逻辑都能按状态分流：  
`requestStatus === 'loading'` 时显示 loading；`dataState === 'empty'` 时显示空态；`uiMode === 'readonly'` 时隐藏编辑和替换按钮。

```tsx
<AIResultCard
  requestStatus={requestStatus}
  uiMode={uiMode}
  dataState={dataState}
  onCopy={handleCopy}
  onReplace={handleReplace}
  onRetry={handleRetry}
/>
```

---

## 三、状态拆分：把交互状态、请求状态、展示状态分开，才好复用

### 3.1 不拆分，就会把视觉、逻辑和数据绑死

很多 AI 组件失败，不是功能做不了，而是“一个状态控制所有东西”。比如 `AI生成弹窗` 点击“开始生成”后，输入框是否继续可编辑、历史结果能否查看、取消按钮是否可用、结果是否先显示占位骨架，这些都不是同一个状态能解决的。

### 3.2 推荐的拆分规则

把状态拆成三层，基本就够用了：

1. **交互状态**：用户当前能不能点、能不能编辑、有没有焦点  
2. **请求状态**：数据是否在请求、成功、失败或空闲  
3. **展示状态**：内容是空、部分填充，还是完整可用

以 `AI结果卡片` 为例：

- 交互状态：是否可点击、是否支持展开、是否显示操作按钮
- 请求状态：是否在重新生成、是否出错
- 展示状态：内容是否为空、是否为摘要、是否为完整结果

拆分后，样式也更容易跟着变化：

- 请求中：右上角显示 loading，内容区保留原内容或显示骨架
- 展示为空：显示空态提示
- 展示完整：显示复制、替换、插入等按钮

### 3.3 可执行的状态机写法

不要在组件里散落条件判断，建议直接把状态机写成统一字段，再映射到 UI。

```ts
type RequestStatus = 'idle' | 'loading' | 'success' | 'error';
type UiMode = 'editable' | 'readonly' | 'preview';
type DataState = 'empty' | 'partial' | 'filled';

const view = {
  loading: requestStatus === 'loading',
  empty: dataState === 'empty',
  disabled: requestStatus === 'loading' || uiMode === 'readonly',
};
```

```tsx
const { requestStatus, dataState, actions, viewProps } = useAIRecommendationPanel({
  sourceText,
  onApply: handleApply,
});
```

这类 Hook 封装的好处是：页面只负责渲染，不用自己拼条件判断。`AI建议面板`、`AI输入框`、`AI生成弹窗` 都可以复用同样的状态组织方式。

---

## 四、主题定制、尺寸变体、密度控制与响应式适配

### 4.1 主题定制：只开放必要入口

主题不是让业务随便改颜色，而是让 AI 组件适配不同产品线。建议只提供有限能力：

- `themeToken`：接入全局 token
- `accentColor`：仅允许强调色定制
- `borderStyle`：特定场景允许替换
- `darkMode`：跟随系统或全局配置

例如 `AI文本润色按钮` 可以允许传入 `accentColor`，但不建议直接开放整个 `style` 系统，否则同一套组件在不同页面会越来越不统一。

### 4.2 尺寸变体与密度控制

AI 组件通常会同时出现在编辑区、侧边栏和弹窗中，空间差异很大。建议提供：

- `size="sm | md | lg"`
- `density="compact | normal"`
- `fullWidth`
- `minHeight`

例如：

- `AI文本润色按钮` 在编辑器工具栏里用 `sm`
- `AI输入框` 在独立页面里用 `lg`
- `AI建议面板` 在侧栏里用 `compact`
- `AI结果卡片` 在列表页里设置固定 `minHeight`，避免内容长短不同导致抖动

```tsx
<AIIdeaPanel size="md" density="compact" fullWidth />
```

### 4.3 响应式适配不要只靠媒体查询

更实用的方式是让组件本身先支持布局收缩，再由页面决定何时折叠。

例如：

- 小屏下 `AI结果卡片` 自动把操作按钮收进更多菜单
- `AI生成弹窗` 在窄屏下改为底部抽屉式
- `AI建议面板` 在移动端只显示最关键建议项
- `AI输入框` 在小尺寸下隐藏次级提示，只保留主输入区

这样组件不是只在理想尺寸里能用，而是在真实页面里也能稳定工作。

---

## 五、加载与禁用态的统一规范：AI 输入框、建议面板、润色按钮怎么落地

### 5.1 AI 输入框

`AI输入框` 最容易出现的问题是：输入中、请求中、不可编辑三种状态混在一起。建议这样约定：

- `disabled=true`：完全禁用，不能输入，不能提交
- `requestStatus='loading'`：可保留输入值，但锁定提交按钮
- `readonly=true`：可查看，不可编辑
- `errorText`：只描述当前输入或请求错误，不替代全局 toast

```tsx
<AIInput
  value={value}
  disabled={disabled}
  requestStatus={requestStatus}
  errorText={errorText}
  onChange={setValue}
  onSubmit={handleSubmit}
/>
```

### 5.2 AI 建议面板

`AI建议面板` 要避免“空白一块”。建议固定保留标题、结果区和操作区：

- 加载时显示骨架，不改整体高度
- 空态显示说明文案
- 错误态提供 `onRetry`
- 成功态展示建议列表，并支持 `onApply`

```tsx
<AIRecommendationPanel
  requestStatus={requestStatus}
  onRetry={handleRetry}
  onApply={handleApply}
>
  {children}
</AIRecommendationPanel>
```

### 5.3 AI 文本润色按钮与 AI 生成弹窗

`AI文本润色按钮` 适合用最小状态集：

- 没选中文本：禁用
- 请求中：显示 loading
- 请求失败：保留按钮，旁边给提示
- 请求成功：触发结果展示或插入

```tsx
<AIPolishButton
  disabled={!selectedText || requestStatus === 'loading'}
  requestStatus={requestStatus}
  onClick={handlePolish}
>
  文本润色
</AIPolishButton>
```

`AI生成弹窗` 则更需要状态机：

- `idle`：展示输入和说明
- `loading`：锁定主操作，保留取消
- `success`：显示结果卡片
- `error`：显示错误提示和重试按钮

```tsx
<AIGenerateDialog
  open={open}
  requestStatus={requestStatus}
  onConfirm={handleGenerate}
  onRetry={handleRetry}
  onCancel={handleClose}
  footer={footer}
/>
```

## 六、文档化与可测试性：把视觉和状态约定写给别人看

如果组件要在多个业务页面复用，文档不能只写“怎么引入”，还要写清楚状态和样式边界。建议每个 AI 组件都配一份轻量模板：

```md
## AIInput

### Props
| name | type | default | description |
| --- | --- | --- | --- |
| value | string | '' | 输入值 |
| size | sm/md/lg | md | 尺寸 |
| disabled | boolean | false | 禁用态 |
| token | object | - | 主题 token |

### Events
| name | payload | description |
| --- | --- | --- |
| onChange | string | 输入变化 |
| onSubmit | string | 触发生成 |
| onError | error | 请求失败 |

### Slots/Children
- suffix：右侧辅助内容
- footer：底部操作区

### 状态
- idle / loading / success / error
- empty / partial / filled
```

有了这个模板，`AI建议面板`、`AI文本润色按钮`、`AI生成弹窗`、`AI结果卡片` 都可以用同一套文档结构表达，页面复用时也更容易对齐。

同时要补上**可测试性**。  
比如 `AI生成弹窗` 的 loading、错误、成功三种状态，最好都能通过测试用例直接驱动；`AI结果卡片` 的按钮显隐、插槽渲染、禁用态文案，也应该能稳定断言。

最小测试清单可以这样写：

- `requestStatus=loading` 时按钮禁用且显示 loading 文案
- `dataState=empty` 时展示空态提示
- `uiMode=readonly` 时隐藏编辑入口
- `children` 传入时正确渲染额外操作区
- `onRetry` 触发后状态回到 `loading`

只有这样，视觉一致性才不是“看起来没问题”，而是“改了也不容易坏”。

---

## 七、在多个业务页面复用时，如何保持一致而不僵化

`AI输入框`、`AI建议面板`、`AI文本润色按钮`、`AI生成弹窗`、`AI结果卡片` 在不同业务页里，通常会有三类差异：

- 页面入口不同：编辑页、详情页、列表页
- 内容密度不同：紧凑、标准、宽松
- 业务插槽不同：提示语、辅助操作、结果处理方式

应对方式不是复制一份新组件，而是通过 `props`、`events`、`children` 控制差异。

- `props` 管基础行为：`size`、`density`、`disabled`、`requestStatus`
- `events` 管业务回调：`onSubmit`、`onApply`、`onRetry`、`onClose`
- `children` 管局部扩展：说明文案、额外按钮、自定义 header/footer

这样页面可以不改组件源码，只改配置就完成复用。  
如果某个页面需要更强的个性化，也应该先尝试增加一个插槽，而不是直接复制出“AI结果卡片 V2”。

---

## 结尾小结：让 AI 组件成为设计系统的一部分

AI 组件能否真正复用，不只看逻辑封装得好不好，更看它能不能**融入现有设计体系**。

把样式接入 token，把状态拆清楚，把加载、禁用、空、错、成功做成统一规范，再通过 props、事件和插槽保留业务差异，AI 能力就不再是某个页面里的临时特效，而是可以在多个业务页面稳定复用的前端资产。

如果再往前一步，就要把这些约定写进**文档化**里：`AI输入框` 怎么配置，`AI建议面板` 什么时候显示，`AI文本润色按钮` 的事件如何回调，`AI生成弹窗` 的状态机如何切换，`AI结果卡片` 的插槽如何扩展，都要在组件说明里讲清楚。这样新同学接手时知道怎么用，老同学改造时知道边界在哪。

最后的行动建议很简单：

1. 先统一 token，不要再写死颜色和间距  
2. 再统一状态机，把交互、请求、展示拆开  
3. 然后补齐 props、events、children 和测试用例  
4. 最后把文档补全，让每个 AI 组件都能按同一套规则发布和维护

只要这四步做到位，AI 组件才算真正进入工程体系。

# 第10章 异步交互封装：请求、取消、重试与兜底策略

# 异步交互封装：请求、取消、重试与兜底策略

## 开篇引入：为什么 AI 能力最容易卡在异步链路上

前端接入 AI 能力后，最先暴露的往往不是“接口通不通”，而是“能不能稳定复用、出了问题怎么收场”。同一个“AI 文本润色”功能，若散落在各处页面里，很快就会变成重复的 `loading / error / success`、各自写请求、重复提交、旧请求覆盖新内容，最终只剩一句“请求失败”。

本质问题不是接口，而是**异步交互没有被抽象成统一能力**。工程化做法不是只封一层请求，而是把它做成一套**状态机 + 可复用 Hook + 可配置交互策略**，让组件层只管展示和触发，异步细节沉到基础层，才能跨页面复用。

---

## 一、异步请求的统一封装思路：先把“状态”抽出来

### 1. 这个问题到底要解决什么？

统一封装的目标不是“把请求写短”，而是让不同组件共享同一套异步语义。AI 场景里，一个动作不只是“发请求并返回结果”，还包括等待、处理中、失败恢复、取消、重试等阶段。

建议把 AI 请求抽象为这些状态：

- `idle`：未触发
- `loading`：请求已发起，正在等待响应
- `processing`：接口已受理，结果仍在生成
- `success`：成功拿到结果
- `error`：请求失败
- `cancelled`：请求被取消
- `retrying`：正在重试

这里要特别区分三个容易混淆的概念：

- `loading`：请求刚发出，通常用于按钮禁用、骨架屏展示；
- `processing`：服务端已开始处理，但还没返回最终内容，适合展示“生成中”或流式内容；
- `disabled`：不是状态，而是 UI 可用性结果，表示当前组件是否允许再次触发。

如果没有这层抽象，组件就会退化成几个布尔值拼状态。表面能跑，真正要支持“取消后恢复”“失败后保留结果”“自动重试一次”时，维护成本会很高。

### 2. 推荐的封装边界

建议把 AI 能力拆成三层：

- **请求层**：只负责调用接口、组装参数、返回数据；
- **状态层**：维护状态机、取消、重试、防抖、节流；
- **展示层**：AI 输入框、建议面板、结果卡片、生成弹窗等 UI。

也就是说，组件不要直接写网络细节，而是通过 Hook 获取统一能力：

```ts
const {
  status,
  data,
  error,
  run,
  cancel,
  retry,
  reset,
  isLoading,
  isDisabled,
  canRetry
} = useAIRequest(options);
```

这个 Hook 对外暴露的是行为和状态，而不是接口实现。这样同一套能力既能服务润色按钮，也能服务建议面板和生成弹窗。

### 3. 设计原则

- **职责单一**：Hook 管流程，组件管界面；
- **状态显式**：不要让一个 `loading` 承担所有阶段；
- **状态拆分**：把“请求中”“生成中”“结果可用”“错误可恢复”分开；
- **可配置**：支持自动触发、手动触发、重试次数、超时时间；
- **可复用**：同一套能力覆盖“润色”“生成”“建议”“摘要”等场景；
- **视觉一致性**：不同组件的加载态、禁用态、错误态保持一致；
- **可测试性**：状态转换、取消、重试都要可单测；
- **文档化**：让别的页面不靠猜就能接入。

---

## 二、统一异步链路示例：把状态机跑完整

一条完整链路可以是：用户在 **AI 输入框** 输入商品标题，触发 **AI 建议面板**，点选建议后进入 **AI 文本润色按钮**，最后在 **AI 生成弹窗** 中确认生成，并用 **AI 结果卡片** 展示结果。

```ts
type AIRequestStatus =
  | 'idle'
  | 'loading'
  | 'processing'
  | 'success'
  | 'error'
  | 'cancelled'
  | 'retrying';

type AIRequestState<T> = {
  status: AIRequestStatus;
  data: T | null;
  error: null | {
    message: string;
    code?: string;
    recoverable?: boolean;
  };
  requestId: number;
};
```

状态转换建议保持清晰：

- `idle → loading → processing → success`
- `idle → loading → error → retrying → success`
- `loading / processing → cancelled → idle`

UI 上也要同步体现：

- `loading`：按钮禁用，显示“请求中...”
- `processing`：结果区显示骨架屏或“生成中...”
- `error`：展示错误提示和重试按钮
- `success`：展示结果卡片，支持复制、接受或继续编辑
- `cancelled`：返回可操作态，不打断当前草稿

如果支持流式返回，`processing` 可以持续更新结果内容，但状态仍保持不变，避免组件误判为“已完成”。

---

## 三、取消请求、重复提交、防抖节流：把边界控制住

### 1. 取消请求：哪些场景必须做？

AI 场景里，取消机制几乎是标配。常见场景包括：

- 用户点击“取消生成”；
- 输入内容变化后终止上一轮请求；
- 弹窗关闭时自动取消未完成任务；
- 路由切换时清理未完成请求；
- 组件卸载时避免向已销毁状态写入数据。

实现上建议把 `AbortController` 或同类能力封装在 Hook 内，组件只拿到 `cancel()`。

### 2. 重复提交：用 UI 和逻辑双层拦截

对于 **AI 文本润色按钮**、**AI 生成弹窗** 这类场景，用户很容易连点。建议同时做两层控制：

- **UI 层禁用**：`isLoading` 或 `isProcessing` 时按钮置灰；
- **逻辑层拦截**：如果已有相同参数请求在执行，就忽略或复用结果。

```tsx
<AIRewriteButton
  text={content}
  loadingText="润色中..."
  disabled={isSubmitting || ai.isDisabled}
  onClick={() => ai.run({ text: content })}
/>
```

按钮文案、禁用态、点击行为要统一，不要只关掉点击却不改视觉状态，否则用户会反复误触。这里的 `disabled` 不是“页面不能点”，而是“当前动作不允许再次触发”，因此要和 `loading / processing` 明确区分。

### 3. 防抖和节流：按场景选，不要乱用

- **防抖**：输入变化后，等用户停下来再触发，适合 **AI 输入框** 和 **AI 建议面板**；
- **节流**：限制单位时间最多触发一次，适合“立即润色”“重新生成”这类按钮操作。

```ts
const debouncedRun = useMemo(() => debounce(ai.run, 500), [ai.run]);
```

输入联想适合防抖，因为用户往往还在改字；按钮点击更适合禁用态 + 拦截，因为它代表明确动作，不需要等待输入稳定。若场景中存在高频更新，还可以在 Hook 层加上请求合并：新参数与旧参数一致时直接复用上一次结果。

### 4. 这里要注意的坑

- 旧请求返回时，是否能识别它已经不是当前上下文；
- 取消后状态是否能回到可操作态；
- 重试是否继承上一次参数；
- 组件卸载后是否还会写入已销毁状态；
- `processing`、`loading`、`disabled` 是否被混用，导致按钮和结果区一起乱跳。

---

## 四、错误态和兜底策略：失败了也要能继续用

### 1. 错误态不是“显示报错”这么简单

AI 能力的错误态，不能只显示红色提示，而要回答三个问题：

1. 发生了什么；
2. 用户现在能做什么；
3. 系统是否有可用替代方案。

因此建议把错误策略拆成三层：

- **提示文案**：告诉用户是网络问题、超时、权限问题还是服务不可用；
- **降级结果**：无法生成时，保留历史结果或提供规则化建议；
- **人工接管**：允许用户手动编辑、复制内容、继续下一步。

### 2. 三类兜底策略怎么选

#### 1）提示文案兜底
适合轻量场景，目标是让用户知道“不是你操作错了”：

- “生成失败，请稍后重试”
- “当前建议暂不可用，已保留原文”
- “网络中断，未影响已填写内容”

#### 2）降级结果兜底
适合需要维持业务链路的场景：

- **AI 建议面板** 没有返回时，展示最近一次结果；
- **AI 文本润色按钮** 失败时，保留原文并给出“复制”“手动修改”；
- **AI 生成弹窗** 失败时，给出结构化模板而不是空白页；
- **AI 结果卡片** 可展示旧结果，并标记“当前内容未刷新”。

#### 3）人工接管兜底
适合结果必须可交付的场景：

- 结果卡片支持编辑；
- 失败后允许用户保存草稿；
- 提供“稍后重试”“转人工处理”等入口。

### 3. 错误对象建议长什么样？

```ts
type AIError = {
  message: string;
  code: string;
  recoverable: boolean;
  fallbackData?: unknown;
  source?: 'request' | 'timeout' | 'cancel' | 'parse' | 'server';
};
```

这样组件就能根据错误类型决定显示什么，而不是统一渲染同一套错误提示。比如 `recoverable=true` 时展示重试按钮，`recoverable=false` 时展示人工接管入口。

---

## 五、失败重试的 UI 骨架：按钮、文案、次数都要定死

很多异步封装只写了 `retry()`，但没有定义重试时 UI 怎么表现，结果就是页面在失败时“看起来能重试，实际上用户不知道怎么操作”。

建议给重试态补一套固定骨架：

```tsx
<AIResultCard
  status={ai.status}
  data={ai.data}
  error={ai.error}
  retryCount={ai.retryCount}
  maxRetryCount={2}
  retryText={ai.status === 'retrying' ? '重试中...' : '重试'}
  retryDisabled={!ai.canRetry}
  onRetry={ai.retry}
/>
```

重试 UI 需要明确：

- **按钮禁用**：到达最大次数后禁用；
- **loading 文案**：重试时显示“重试中...”；
- **次数限制**：默认 1～2 次，避免无限循环；
- **错误保留**：重试前不要清空错误信息，成功后再统一重置；
- **参数继承**：重试默认沿用上一次入参，不让用户重复填写。

这些细节会直接影响体验一致性，也是 AI 组件能否真正落地的关键。

---

## 六、把异步逻辑沉淀为 Hook：useAIRequest / useAIAction

### 1. 为什么要用 Hook 而不是每个组件各写一套？

因为 AI 能力往往横跨多个组件：

- **AI 输入框**：输入时自动请求建议；
- **AI 建议面板**：展示建议并允许刷新；
- **AI 文本润色按钮**：点击后生成润色结果；
- **AI 生成弹窗**：提交参数后返回生成内容；
- **AI 结果卡片**：展示最终结果并支持重试、复制、接受。

如果每个组件都自己写请求、取消、错误、重试，就会出现逻辑重复、状态不一致、测试困难。

### 2. 一个可复用 Hook 应该提供什么

建议把 Hook 设计成“动作 + 状态 + 派生值”三类输出：

```ts
const ai = useAIAction({
  request: api.generateText,
  auto: false,
  debounce: 300,
  retry: { count: 2, interval: 800 },
  onSuccess(data) {},
  onError(err) {}
});
```

输出可以包含：

- `status`：当前状态；
- `data`：成功结果；
- `error`：失败信息；
- `run(params)`：发起请求；
- `cancel()`：取消请求；
- `retry()`：重试上一次请求；
- `reset()`：重置状态；
- `isLoading / isDisabled / canRetry`：供 UI 直接使用。

### 3. 一个更完整的实现骨架

```ts
function useAIRequest<TParams, TResult>(options: {
  request: (params: TParams, signal?: AbortSignal) => Promise<TResult>;
  retryCount?: number;
}) {
  const [status, setStatus] = useState<AIRequestStatus>('idle');
  const [data, setData] = useState<TResult | null>(null);
  const [error, setError] = useState<AIError | null>(null);
  const controllerRef = useRef<AbortController | null>(null);
  const lastParamsRef = useRef<TParams | null>(null);

  const cancel = useCallback(() => {
    controllerRef.current?.abort();
    setStatus('cancelled');
  }, []);

  const run = useCallback(async (params: TParams) => {
    lastParamsRef.current = params;
    controllerRef.current?.abort();
    controllerRef.current = new AbortController();

    setStatus('loading');
    setError(null);

    try {
      const result = await options.request(params, controllerRef.current.signal);
      setStatus('success');
      setData(result);
      return result;
    } catch (e) {
      if (controllerRef.current.signal.aborted) {
        setStatus('cancelled');
        return;
      }
      setStatus('error');
      setError({
        message: '生成失败，请稍后重试',
        code: 'AI_REQUEST_FAILED',
        recoverable: true
      });
    }
  }, [options]);

  const retry = useCallback(() => {
    if (!lastParamsRef.current) return;
    setStatus('retrying');
    return run(lastParamsRef.current);
  }, [run]);

  const reset = useCallback(() => {
    setStatus('idle');
    setError(null);
  }, []);

  return {
    status,
    data,
    error,
    run,
    retry,
    cancel,
    reset,
    isLoading: status === 'loading' || status === 'processing' || status === 'retrying',
    isDisabled: status === 'loading' || status === 'processing',
    canRetry: status === 'error'
  };
}
```

这个 Hook 的价值在于：**把异步请求、取消、重试、错误兜底统一收口**，组件只消费状态和事件，不再关心底层流程。

### 4. 事件也要统一命名

Hook 或组件的事件，建议保持稳定：

- `onSuccess(result)`：成功返回；
- `onError(error)`：失败返回；
- `onCancel()`：取消完成；
- `onRetry()`：触发重试；
- `onStateChange(status)`：状态切换。

有了统一事件，**AI 输入框**、**AI 建议面板**、**AI 文本润色按钮**、**AI 生成弹窗**、**AI 结果卡片** 才能共用一套接入方式。

---

## 七、可测试性：异步封装能不能测，决定它能不能长期维护

异步逻辑最好不要和 UI 强耦合。建议把状态机和请求逻辑单独抽出来测，至少覆盖以下路径：

- 输入参数是否正确传递；
- 状态转换是否符合预期；
- 取消后是否不会再写入结果；
- 重试是否继承上一次参数；
- 错误态是否能正确展示兜底内容；
- `isDisabled` 是否在 `loading / processing` 时准确生效。

如果是组件测试，可以重点验证：

- **AI 输入框** 输入变化后是否触发防抖请求；
- **AI 建议面板** 的空态、加载态、错误态是否正确；
- **AI 文本润色按钮** 是否在加载时禁用；
- **AI 生成弹窗** 关闭时是否取消请求；
- **AI 结果卡片** 是否在失败后保留旧结果并提供重试。

这些测试最好围绕状态机断言，而不是只看最终 DOM 文案。这样一旦以后新增状态，比如 `paused`、`quotaExceeded`，也更容易扩展。

---

## 八、文档化：让同事可以直接接入，而不是靠猜

如果希望别的页面直接复用 AI 组件，文档必须写清楚：

- 适用场景；
- `props` 定义；
- 事件回调；
- 插槽/`children` 用法；
- 状态机说明；
- 错误与兜底策略。

可以按这个模板写 README：

| 项目 | 说明 |
| --- | --- |
| 组件名称 | `AIRewriteButton` |
| 适用场景 | 文本润色、内容改写、语气优化 |
| 核心 props | `text`、`prompt`、`loadingText`、`disabled` |
| 核心事件 | `onSuccess`、`onError`、`onCancel` |
| 支持插槽/children | 支持自定义结果展示和按钮文案 |
| 状态机 | `idle/loading/processing/success/error/cancelled/retrying` |
| 兜底策略 | 错误提示、保留原文、手动编辑、重试 |

再补一段最小示例：

```tsx
<AIRewriteButton
  text={content}
  prompt="请保持原意，优化表达"
  loadingText="润色中..."
  disabled={isSubmitting}
  onSuccess={(result) => setContent(result)}
  onError={(err) => toast(err.message)}
/>
```

如果是更复杂的 **AI 生成弹窗**，还可以在文档里补充 `children` 示例，让业务方能替换弹窗底部按钮、说明文案或结果预览区域：

```tsx
<AIGenerateModal
  open={open}
  title="生成商品卖点"
  onConfirm={ai.run}
  onCancel={ai.cancel}
>
  <div slot="header">支持输入商品标题和风格要求</div>
  <div slot="footer">
    <button onClick={ai.cancel}>取消</button>
    <button onClick={ai.run}>开始生成</button>
  </div>
</AIGenerateModal>
```

文档做到这一步，组件才算真正可以复用。

---

## 九、发布与维护：让 AI 异步能力长期可用

AI 异步封装做出来之后，真正的工作才刚开始。后续发布与维护，重点看这几个方面：

- **版本兼容**：状态枚举、事件名、参数结构尽量稳定；
- **测试覆盖**：至少覆盖成功、失败、取消、重试、卸载五类路径；
- **日志可观测**：关键错误码、取消原因、重试次数要能追踪；
- **文档同步**：新增可配置项后，示例与说明要一起更新；
- **默认策略清晰**：比如默认是否自动请求、默认重试几次、默认是否保留旧结果；
- **可配置项收口**：把超时、重试间隔、是否允许取消、是否自动降级都作为统一配置暴露出来。

如果要升级协议，建议提前列一个变更清单：

1. 是否新增状态；
2. 是否改变事件签名；
3. 是否影响默认重试逻辑；
4. 是否影响错误兜底文案；
5. 是否需要同步更新 README 和测试用例。

---

## 结尾小结：把异步链路做成资产，而不是散点代码

AI 前端能力能不能在工程里长期活下来，关键看异步链路有没有被真正抽象。你需要记住四个落点：

1. **统一状态机**：不要只剩一个 `loading`，要把 `loading / processing / disabled` 区分开；
2. **取消与重试可控**：防止旧请求污染新结果；
3. **错误有兜底**：给用户继续操作的路径；
4. **Hook 沉淀复用**：让 AI 输入框、建议面板、文本润色按钮、生成弹窗、结果卡片共享同一套逻辑。

如果准备在项目里落地，建议按这个顺序推进：

- 先定义状态枚举；
- 再封装请求、取消、重试；
- 然后补齐错误与降级策略；
- 最后抽成 `useAIRequest` 或 `useAIAction`，接入多个组件；
- 同步补文档、补测试、补默认配置。

这样做完之后，AI 能力就不再是某个页面上的临时逻辑，而是可以被反复调用、持续维护的前端资产。

# 第11章 Hook 封装：把 AI 能力从组件里再抽一层

# Hook 封装：把 AI 能力从组件里再抽一层

## 开篇引入：为什么组件已经不够了？

当你把“AI 建议”“AI 润色”“AI 生成内容”做成组件后，第一版通常能很快落地：输入框、按钮、结果卡片、弹窗直接组合，页面马上可用。  
但一旦业务增多，**请求逻辑、加载状态、错误兜底、缓存命中、重复点击防抖**就会在多个组件里重复出现。

如果继续把这些逻辑写死在组件里，结果往往是：

- 组件越来越胖，UI 和业务逻辑混在一起
- 不同页面的 AI 行为不一致
- 一个页面改了请求规则，别的页面没同步
- 测试和文档都难维护

更合理的做法是：**UI 留在组件，AI 交互规则收进 Hook**。  
这样页面和容器可以变，但底层行为保持一致，AI 能力也才能真正沉淀成可复用资产。

---

## 一、Hook 适合承载哪些 AI 逻辑？

### 1. 适合抽到 Hook 里的逻辑清单

Hook 适合承载的是那些**跨组件复用、与 UI 弱相关**的逻辑，典型包括：

- 请求发起、重试、取消
- 输入内容整理、参数组装
- 状态机切换：`idle / loading / success / error / disabled`
- 缓存命中与过期判断
- 错误兜底和降级文案
- 事件派发：开始、成功、失败、结束
- 派生状态：是否可提交、是否可点击、是否显示建议面板

例如“AI 文本润色按钮”会出现在详情页、编辑页、弹窗里，但润色请求、loading、错误提示、结果回填的规则基本一致，这部分就很适合放进 Hook。

### 2. 不建议进 Hook 的内容

更适合留在组件层的，是：

- 具体弹窗布局
- 某个页面专属动画
- 复杂 DOM 测量
- 特定业务文案拼装
- 强依赖页面结构的联动逻辑

Hook 的价值是复用规则，不是替代 UI。  
如果把强视觉逻辑也塞进去，Hook 很快会变成“看不见的组件”，调用方反而搞不清它负责什么。

### 3. 判断标准

一个实用标准是：

> 这段逻辑是否能独立服务两个以上不同页面？

能，就优先 Hook 化。  
不能，就先留在组件里，不要为了抽象而抽象。

---

## 二、请求状态、缓存状态、错误状态如何拆分？

### 1. 请求状态：描述动作是否进行中

请求状态用于控制交互节奏，常见字段：

- `loading`：是否正在请求
- `submitting`：是否正在提交
- `pending`：是否有未完成任务
- `disabled`：是否禁止再次触发

例如在 AI 输入框里，用户点击“生成建议”后，输入框可以继续编辑，但按钮应进入 `loading` 并短暂禁用；在 AI 生成弹窗里，确认按钮进入 `loading` 后，通常还要禁用关闭或重复提交。

### 2. 缓存状态：减少重复请求

缓存状态适合承载：

- 最近一次请求结果
- 相同输入是否已命中
- 是否允许直接复用上次结果
- 缓存过期时间

比如 AI 建议面板，在输入内容不变时就没必要再次发请求；AI 结果卡片在同一字段多次打开时，也可以直接展示缓存，减少等待。

### 3. 错误状态：不要只返回一个 message

错误状态建议至少拆成：

- `errorCode`
- `errorMessage`
- `retryable`
- `fallbackData`

这样组件才能判断：是显示提示、给出重试按钮，还是直接展示兜底内容。  
只返回一个字符串，最后 UI 往往只能“报错一刀切”。

---

## 三、三类 Hook：useAIAction、useAIRequest、useAIState 怎么分工？

### 1. useAIAction：负责“触发一个 AI 动作”

它适合“AI 文本润色按钮”“AI 生成弹窗确认”这类一次性动作。

**职责边界：**

- 接收触发参数
- 控制按钮 loading
- 处理成功、失败、结束回调
- 暴露 `run / reset / retry / cancel`

**推荐返回值结构：**

```ts
{
  data,
  status,
  error,
  loading,
  canRun,
  run,
  retry,
  reset,
  cancel,
  hasData
}
```

**适合场景：**

- 点击“润色一下”
- 点击“生成标题”
- 弹窗里点“开始生成”

组件只管把按钮、图标、tooltip、文案渲染出来；真正的异步执行交给 `useAIAction`。

### 2. useAIRequest：负责“标准化请求生命周期”

它更偏底层，是通用请求引擎，适合把所有 AI 异步调用统一起来。

**职责边界：**

- `execute(params)`
- `abort()`
- `status`
- `data`
- `error`
- `lastParams`
- `refresh()`

**推荐返回值结构：**

```ts
{
  data,
  status,
  error,
  loading,
  run: execute,
  retry: refresh,
  reset,
  cancel: abort,
  lastParams
}
```

**适合场景：**

- AI 建议面板拉取推荐项
- AI 结果卡片刷新内容
- AI 输入框发起联想查询

可以把它理解成“通用请求层”，上层的 `useAIAction` 也可以基于它组合，不必每个页面都自己写一套 loading、错误和重试逻辑。

### 3. useAIState：负责“共享 AI 业务态”

当多个组件需要共享同一份 AI 状态时，就该用 `useAIState`。

**职责边界：**

- 当前模式：建议、润色、生成、预览
- 当前输入源
- 当前结果
- 是否展示面板
- 是否允许编辑
- 当前选中项

**推荐返回值结构：**

```ts
{
  mode,
  inputValue,
  result,
  panelVisible,
  editable,
  selectedItem,
  setMode,
  setInputValue,
  setResult,
  openPanel,
  closePanel
}
```

**适合场景：**

- 一个页面同时有 AI 输入框和 AI 建议面板
- AI 结果卡片与生成弹窗联动
- 多个入口共用同一份“商品标题优化”状态

它解决的是“多个组件看同一份状态”的问题，而不是单次请求的问题。

---

## 四、状态拆分：展示状态、请求状态、交互状态分开写

很多 AI 组件出问题，不是因为接口复杂，而是状态混在一起。建议至少拆成三类。

### 1. 展示状态

只负责“页面上看起来是什么样”：

- `panelVisible`
- `showEmpty`
- `showError`
- `showResult`
- `collapsed`

### 2. 请求状态

只负责“异步有没有在跑”：

- `loading`
- `success`
- `error`
- `disabled`
- `pending`

### 3. 交互状态

只负责“用户怎么操作”：

- `inputValue`
- `selectedItem`
- `draftValue`
- `hasEdited`
- `lastAction`

例如 AI 建议面板里，`panelVisible` 变了不代表请求状态变了；输入框内容变了也不代表结果已经失效。把这些状态拆开，组件才不会因为一个布尔值变化而全局重绘。

---

## 五、组件与 Hook 的分工：UI 归组件，流程归 Hook

### 1. 组件负责什么？

组件负责：

- 视觉布局
- 交互入口
- 插槽 / children 渲染
- 状态展示
- 文案和样式统一

例如：

- **AI 输入框**：展示输入区、快捷触发、辅助提示
- **AI 建议面板**：负责列表展示、选中态、空状态
- **AI 文本润色按钮**：负责按钮样式、icon、tooltip
- **AI 生成弹窗**：负责弹层结构、表单布局、确认区
- **AI 结果卡片**：负责结果展示、复制、重新生成入口

### 2. Hook 负责什么？

Hook 负责：

- 调用规则
- 状态机切换
- 异步调度
- 缓存策略
- 错误兜底
- 可复用事件处理

### 3. 用 children 承接差异化 UI

当页面之间只差一点 UI 时，不要复制 Hook，也不要复制组件。  
更好的方式是：Hook 作为逻辑底座，把差异 UI 放进 `children` 或插槽。

```tsx
<AIActionPanel title="AI 润色" onRun={run} loading={loading}>
  {({ result, error }) => (
    <>
      {result && <AIResultCard data={result} />}
      {error && <ErrorTip text={error.message} />}
    </>
  )}
</AIActionPanel>
```

这样就实现了“逻辑统一，展示可变”。

---

## 六、状态机要怎么写，组件才好消费？

### 1. 不要只写一个状态字段

AI 组件最好明确状态机，而不是靠多个布尔值互相猜。  
推荐最少包含：

- `idle`
- `loading`
- `success`
- `error`
- `disabled`

例如 AI 生成弹窗：

- `idle`：等待用户填写参数
- `loading`：正在生成
- `success`：展示结果并允许确认
- `error`：展示错误和重试
- `disabled`：权限不足或频次限制

### 2. 状态机和事件搭配

建议把事件也标准化：

- `onStart`
- `onSuccess`
- `onError`
- `onRetry`
- `onReset`

这样组件只需要订阅事件，不需要自己猜测状态变化点。

### 3. 状态机如何驱动 UI？

- `loading`：按钮禁用，显示骨架屏
- `success`：渲染 AI 结果卡片
- `error`：渲染错误提示与重试按钮
- `idle`：显示占位文案
- `disabled`：整体灰化并解释原因

---

## 七、Hook 化后，如何处理加载、禁用和错误兜底？

### 1. 加载态：让用户知道“正在处理”

建议统一约定：

- 请求中按钮进入 `loading`
- 输入区保持可编辑，或按业务决定是否禁用
- 面板内显示骨架屏或占位文案

如果没有加载态，用户会以为按钮没点上，于是连点多次，前后端都容易出问题。

### 2. 禁用态：避免重复触发

常见禁用条件有：

- 输入为空
- 请求正在进行
- 当前数据未变化
- 权限不足
- 达到频次限制

最好把这些规则收进 Hook，组件只看 `canRun`，不要每个页面都手写判断。  
这样既减少出错，也能保证不同页面的交互一致。

### 3. 错误兜底：不要让页面“空白失败”

至少准备三层兜底：

- 轻错误：提示重试
- 中错误：保留历史结果
- 重错误：降级为普通文本或隐藏 AI 区块

例如 AI 生成弹窗失败后，仍允许用户继续手动编辑；AI 结果卡片失败后，保留上一次结果并标注“内容可能未刷新”。  
这样用户不会因为一次失败就丢掉所有上下文。

## 八、可直接复用的 Hook 代码骨架

下面以“AI 文本润色按钮 + 结果回填”为例，给一个可落地的骨架。

```ts
type AIActionOptions<TInput, TResult> = {
  request: (input: TInput, signal?: AbortSignal) => Promise<TResult>
  onSuccess?: (result: TResult) => void
  onError?: (error: unknown) => void
  cacheKey?: (input: TInput) => string
  enabled?: boolean
}

export function useAIAction<TInput, TResult>({
  request,
  onSuccess,
  onError,
  cacheKey,
  enabled = true,
}: AIActionOptions<TInput, TResult>) {
  const [status, setStatus] = useState<'idle' | 'loading' | 'success' | 'error'>('idle')
  const [data, setData] = useState<TResult | null>(null)
  const [error, setError] = useState<unknown>(null)
  const [lastInput, setLastInput] = useState<TInput | null>(null)
  const cacheRef = useRef(new Map<string, TResult>())

  const loading = status === 'loading'
  const canRun = enabled && !loading

  const run = useCallback(async (input: TInput) => {
    if (!canRun) return

    const key = cacheKey?.(input)
    if (key && cacheRef.current.has(key)) {
      const cached = cacheRef.current.get(key)!
      setData(cached)
      setStatus('success')
      onSuccess?.(cached)
      return cached
    }

    setStatus('loading')
    setError(null)
    setLastInput(input)

    try {
      const result = await request(input)
      setData(result)
      setStatus('success')
      if (key) cacheRef.current.set(key, result)
      onSuccess?.(result)
      return result
    } catch (e) {
      setError(e)
      setStatus('error')
      onError?.(e)
      throw e
    }
  }, [cacheKey, canRun, onError, onSuccess, request])

  const retry = useCallback(() => {
    if (lastInput == null) return
    return run(lastInput)
  }, [lastInput, run])

  const reset = useCallback(() => {
    setStatus('idle')
    setData(null)
    setError(null)
  }, [])

  const cancel = useCallback(() => {
    reset()
  }, [reset])

  return {
    data,
    error,
    status,
    loading,
    canRun,
    run,
    retry,
    reset,
    cancel,
    hasData: data != null,
  }
}
```

这个骨架里已经包含了几个关键点：

- **状态拆分**：`status / data / error`
- **派生值**：`loading / canRun / hasData`
- **缓存状态**：`cacheRef`
- **事件回调**：`onSuccess / onError`
- **重试能力**：`retry`
- **兜底能力**：`reset / cancel`

组件里只需要把 UI 绑定到这些输出上，不必再重复写请求和判断逻辑。

## 九、可测试性与文档化：Hook 比组件更适合做单测

### 1. 为什么 Hook 更适合测试？

Hook 的职责比组件更稳定，测试也更容易聚焦状态流转。  
建议重点覆盖这些场景：

- 请求成功时状态是否正确
- 请求失败时错误是否落到位
- `retry` 是否使用上次参数
- `canRun` 是否随输入变化
- `cancel` 是否能清理 pending
- 缓存命中时是否跳过重复请求

如果状态机设计清楚，测试也更容易写。  
在 AI 场景里，测试不要只看“有没有渲染出来”，更要看状态流转是否符合预期。

### 2. 文档化要写什么？

文档不要只写“怎么用”，而要把边界说清楚：

- 适用场景
- 输入参数说明
- 输出字段说明
- 事件回调说明
- 状态机流转图
- 默认值与可配置项
- 错误兜底策略

例如 `useAIAction` 的文档，至少要说明：

- 什么时候进入 `loading`
- 什么时候触发 `onSuccess`
- 失败时 `retry` 是否可用
- 是否支持 `cancel`
- 哪些参数可以覆盖默认行为

文档写清楚，别的同学接手时才不会乱用。

---

## 十、在多个业务页面复用时要注意什么？

### 1. 保持能力边界稳定

不要让 Hook 随着某个页面需求不断变胖。  
一旦某个字段只服务单页面，优先通过参数扩展或外层适配解决，不要直接把业务细节写死在 Hook 内。

### 2. 统一命名和状态机

同类 Hook 最好统一命名：

- `run / retry / reset`
- `loading / error / data`
- `idle / loading / success / error`

这样不同页面的学习成本最低，也方便团队形成一致习惯。

### 3. 组件与 Hook 一起发布

真正可复用的，不是单独一个 Hook，而是一组配套能力：

- `useAIRequest` 负责逻辑
- `AIInput` 负责触发
- `AIResultCard` 负责展示
- `AIActionPanel` 负责组合

它们组合起来，才是可以跨页面复用的 AI 能力包。  
如果只有 Hook，没有组件，使用成本会很高；如果只有组件，没有 Hook，底层逻辑还是会重复。

---

## 结尾小结：先抽逻辑，再谈扩展

AI 功能要在前端真正复用，不能只停留在“再写一个按钮组件”。更稳妥的方式是：**组件负责表现，Hook 负责规则**。

你可以按这个顺序落地：

1. 先识别可复用的 AI 交互逻辑  
2. 再拆请求状态、缓存状态、错误状态  
3. 然后设计 `useAIAction / useAIRequest / useAIState`  
4. 再把展示状态、请求状态、交互状态分开  
5. 最后统一输出数据、状态、方法、派生值  
6. 配套写清楚文档、测试和默认兜底  

如果你在“AI 输入框、AI 建议面板、AI 文本润色按钮、AI 生成弹窗、AI 结果卡片”之间频繁复制代码，往往不是组件不够，而是 **Hook 还没抽到位**。

# 文档化、可测试性与发布维护：让 AI 组件真正成为资产

## 开篇引入：为什么“做出来”不等于“沉淀下来”

很多团队把 AI 能力接进组件库后，第一版通常都能跑通：有 **AI 输入框** 收集提示词，有 **AI 建议面板** 输出推荐，有 **AI 文本润色按钮** 做局部改写，复杂一点还会封成 **AI 生成弹窗** 或 **AI 结果卡片**，把生成、展示、操作收口。

但问题往往出在第二步：换页面、换业务线，代码开始分叉。有人重写 loading，有人把错误态直接清空，有人改了 `props` 名字却没同步文档。最后，同一个 AI 能力在不同地方长成不同版本，组件不再是资产，只剩“能跑的业务代码”。

所以，AI 组件真正要解决的不是“能不能接上”，而是“能不能长期复用”。这要求它具备四个特征：**文档清楚、行为可测、发布可控、维护可迭代**。做出来只是起点，沉淀下来才算资产。

## 一、标准组件文档模板：把接入契约写清楚

一个可复用的 AI 组件，README 不能只写“用于 AI 生成”。建议固定成一套模板，让接入方一眼判断能不能用、怎么用、边界在哪。

```md
# AiGenerateModal

## 用途
用于在弹窗中完成 AI 生成、预览、确认插入的完整流程。

## API
见 Props 表格。

## 事件
见 Events 表格。

## 插槽 / children
支持自定义标题、底部操作区、结果区。

## 状态说明
idle / inputting / loading / success / error / disabled

## 注意事项
- loading 时禁止重复提交
- error 时保留输入，便于重试
- disabled 不进入请求流程
```

### Props 表格示例

```md
| prop | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| loading | boolean | false | 是否处于加载中 |
| disabled | boolean | false | 是否禁用交互 |
| mode | 'quick' \| 'advanced' | 'quick' | 交互模式 |
| placeholder | string | '' | 输入框提示文案 |
| maxItems | number | 3 | 建议条数 |
| keepInputOnSuccess | boolean | true | 成功后是否保留输入 |
| requestTimeout | number | 10000 | 请求超时毫秒数 |
```

### 事件表格示例

```md
| 事件名 | 参数 | 触发时机 |
|---|---|---|
| onSubmit | value: string | 用户提交输入时 |
| onGenerateStart | void | 请求开始前 |
| onGenerateSuccess | result: any | 生成成功后 |
| onGenerateError | error: Error | 生成失败后 |
| onRetry | void | 点击重试时 |
```

### 插槽 / children 示例

```jsx
<AiResultCard title="AI 结果" onConfirm={handleInsert}>
  <div slot="description">结果可直接插入正文，也可继续编辑。</div>
</AiResultCard>
```

把用途、API、事件、插槽、状态说明、注意事项写全，文档就不是说明文字，而是可执行的接入契约。

---

## 二、示例驱动的文档：接入、边界、错误都要写

文档最怕只有正常路径，没有边界。建议每个组件至少提供三类示例，并尽量保持同一套命名，方便横向复用。

### 1. 接入示例

```jsx
<AiPolishButton
  value={text}
  onGenerateSuccess={(next) => setText(next)}
/>
```

### 2. 边界示例

```jsx
<AiSuggestPanel
  loading={loading}
  disabled={!keyword}
  maxItems={5}
>
  <span>输入关键词后再生成建议</span>
</AiSuggestPanel>
```

### 3. 错误示例

```jsx
<AiGenerateModal
  error="当前服务繁忙"
  showRetry
  onRetry={handleRetry}
/>
```

这里的 `children` 或插槽很关键：业务页面可以塞自己的说明、按钮、空态文案，但外壳和状态保持统一。这样既保留**视觉一致性**，也避免组件 API 被一堆布尔开关塞满。

## 三、可测试性设计：先拆 Hook，再测状态机

AI 组件要好测，关键是把异步逻辑从 UI 中抽离。最实用的方式就是 **Hook 封装**，比如 `useAiAction` 统一管理请求、取消、重试和状态迁移。

### 1. 单测测什么

单测重点放在 Hook 和状态管理层：

- 调用后是否从 `idle` 进入 `loading`
- 成功后是否进入 `success`
- 失败后是否进入 `error`
- `disabled` 时是否不发起请求
- `requestTimeout` 是否生效

### 2. 交互测试怎么写

以 **AI 文本润色按钮** 为例：

- 点击后按钮是否进入禁用态
- `loading` 时是否重复提交
- 成功后是否把结果回写到输入框
- 失败后是否显示重试入口
- 关闭弹窗后状态是否正确重置

### 3. 状态机测试覆盖哪些异常

不要只测“点了会成功”，还要测状态流转是否合法：

- `idle -> inputting -> loading -> success`
- `idle -> inputting -> loading -> error -> loading -> success`
- `inputting -> disabled`
- `disabled` 下不能进入 `loading`

如果状态机测试过关，多人协作时就不容易把 loading、错误态、禁用态写乱。

## 四、发布维护流程：版本、兼容、灰度、回滚都要有

AI 组件一旦进入多个业务页面，发布就不能“改完就发”。建议建立固定 checklist。

### 发布前 Checklist

- [ ] 是否更新 README
- [ ] 是否补充 Props / Events 表格
- [ ] 是否补充状态机说明
- [ ] 是否补充测试用例
- [ ] 是否记录破坏性变更

### 版本号与变更说明

每次升级都要明确：

- 新增了哪些 `props`
- 哪些事件名变了
- 默认行为有没有调整
- 哪些老接口准备废弃

如果是破坏性变更，尽量按大版本升级；如果只是新增能力，优先保持向后兼容，并在文档里标注清楚。

### 兼容策略与灰度升级

优先保留老接口可用：

- 旧 `props` 先保留兼容层
- 废弃能力用 `deprecate` 提示
- 新旧行为并行一段时间
- 必要时提供适配器组件

先在低风险页面试点，比如内部工具页、实验页，再逐步推广到主流程。灰度期间重点观察：

- 文档是否清楚
- 错误态是否稳定
- `loading` 和 `disabled` 是否合理
- 视觉是否统一

### 回滚预案

如果出现问题，直接回滚到上一个稳定版本，并保留变更记录，避免下次重复踩坑。回滚不是失败，而是资产化维护里必须预留的安全阀。

## 五、多业务页面复用：让文档、测试和维护形成闭环

真正的资产，不是能复制粘贴，而是能在不同页面持续复用。可以拿“**商品标题优化**”做一条贯穿链路：

1. 在商品编辑页放一个 **AI 输入框** 收集标题诉求  
2. 用 **AI 建议面板** 展示多个候选标题  
3. 点击 **AI 文本润色按钮** 对当前标题做局部优化  
4. 需要完整生成时，打开 **AI 生成弹窗**  
5. 结果写入 **AI 结果卡片**，支持复制、插入、重试

这条链路里，文档负责告诉业务怎么接，测试负责保证行为稳定，发布策略负责控制风险。多个页面共用同一套组件后，反馈也会更集中：

- `props` 是否足够直观
- 事件是否稳定
- 状态是否覆盖完整
- 样式是否容易统一

再反向迭代组件和 **Hook 封装**，让下一个业务接入更轻。一个好的资产，不只是“别人能用”，还要“别人用完能反馈回来”。

## 结尾小结：把 AI 组件当成产品来维护

AI 组件不是临时拼装件，而是长期维护的工程资产。你要用文档告诉别人怎么接，用测试保证行为不漂，用发布策略控制风险，用维护机制推动迭代。

最后记住一条检查清单：**文档是否清楚、状态是否拆分、Hook 是否独立、错误是否兜底、测试是否覆盖、发布是否可回滚**。再加上一条更现实的标准：这个组件能不能被第二个、第三个业务页面直接复用。能做到这一点，AI 输入框、AI 建议面板、AI 文本润色按钮、AI 生成弹窗、AI 结果卡片，才真正从功能实现变成了前端资产。
